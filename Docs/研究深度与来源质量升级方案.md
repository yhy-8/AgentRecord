# 研究深度与来源质量升级方案

> 状态：后续候选方案，记录于 2026-07-19，当前不实施。现有检索、Researcher、Reviewer、数据库和报告格式继续作为稳定基线；本文不能用来声称对应能力已经上线。

## 1. 决策与目的

当前第三方搜索流程已经把开放式工具循环收敛为确定性工作流：中控执行固定查询、分配 `W-*`、限制模型接触 URL、复用检索证据并在最后渲染链接。近期应先观察这套流程的真实成功率、成本和失败类型，不继续修改代码。

当前质量边界也需要明确：Researcher 主要依据搜索服务返回的标题、摘要和日期写作，Reviewer 只能在这些材料范围内判断来源是否支持正文；“优先一手和权威来源”目前主要是模型指令，不是中控可证明的硬约束。因此，一次审查通过不等于已经核对网页全文，也不等于来源本身一定是原始、权威来源。

后续升级的目的不是增加更多自由 Agent，而是在保留现有可靠性边界的前提下：

- 从“搜索摘要支持”提升到“可定位的原文证据支持”；
- 让来源权威性、独立性和时效性可以审计；
- 让关键事实与证据建立可检查的对应关系；
- 在提高研究质量时继续限制联网、模型调用、上下文和失败重试成本；
- 保持报告可独立阅读，不让 Markdown 报告依赖数据库才能成立。

## 2. 稳定观察期

开始实现前，建议至少连续观察四个闭合自然周，并包含至少一份月报；同一输入的自动重试和手动重跑只算同一个报告样本，不能用来虚增成功率。期间除安全、数据损坏或明确回归外，不改变报告流水线。

观察直接使用现有日志、Markdown 报告和 SQLite 审计数据，不删除或重建数据库。每个独立报告至少记录：

- 首次运行成功率和经过允许重试后的最终成功率；
- 失败所在阶段、修订次数、Reviewer 否决原因和最终是否交付；
- 总 token、缓存 token、模型耗时、总耗时和搜索次数；
- 每个选题的搜索结果数、实际引用数、零结果和主题丢弃情况；
- 引用来源的域名分布，以及人工抽查的一手、权威、二手、聚合或镜像分类；
- 关键数字、日期、因果判断是否能由链接页面直接核对。

观察结束后形成一份基线报告。没有基线时，不以单次成功或失败决定升级，也不把旧架构和新架构的历史运行直接混为同一成功率。

## 3. 未来来源分级

来源分级用于选择和审查证据，不用于简单封禁整个域名。建议定义：

| 等级 | 定义 | 典型来源 | 使用规则 |
| --- | --- | --- | --- |
| S1 原始来源 | 事实、数据或规则的直接发布者 | 官方公告、法规标准、原始论文、统计机构、公司财报 | 关键事实优先使用 |
| S2 权威二手来源 | 有编辑、审稿或专业责任的可靠解释 | 学术综述、出版社、专业机构报告、严肃行业数据库 | 可支持多数事实，重要数字优先与 S1 互证 |
| S3 可信报道与分析 | 能识别作者、时间和引用链的二手材料 | 可靠媒体、专业媒体、具名专家分析 | 用于背景、争议和时效性信息；关键结论尽量交叉验证 |
| S4 聚合、镜像与弱来源 | 原始出处不清或主要复制、转述内容 | 文档镜像、内容农场、无出处聚合页、匿名帖子 | 只作发现线索或补充视角，不得单独支撑关键事实 |

分级不能只依赖域名白名单。中控还应考虑发布者身份、作者和日期、是否给出原始引用、内容是否为转载、多个页面是否实际来自同一通讯稿，以及主题所属领域。无法获得 S1/S2 时可以保留较弱证据，但必须说明检索边界和不确定性，不能静默把 S4 当成权威来源。

## 4. 候选证据工作流

升级后仍以确定性工作流为主，建议按以下顺序实施：

```text
ResearchPlanner 固定问题与查询
        ↓
中控执行一次发现性搜索
        ↓
URL 规范化、去重、来源初筛与排序
        ↓
有上限地获取少量候选原文
        ↓
正文抽取、来源分级、证据卡与内容哈希
        ↓
Researcher 依据证据卡写作并声明“结论—证据”关系
        ↓
确定性门禁 + Reviewer 语义审查
        ↓
中控渲染真实链接并原子交付
```

### 4.1 发现、排序与原文获取

- 保留当前每个固定查询最多 10 条搜索结果的发现阶段。
- 先去除重复 URL、转载副本和明显无关结果，再按直接性、权威性、时效性、独立性和可访问性排序。
- 只获取每个主题排名靠前的少量候选，不遍历整个结果集。初始建议上限为每个主题 4 个、整份报告 8 个，具体值在观察基线后校准。
- 只在现有证据明确缺少关键事实、反例或边界时允许补充检索；每个主题最多一次，并保存“证据缺口—补充查询—新增证据”的因果记录。格式修订不得触发补充检索。
- 网页无法访问、正文为空或只有转载时，按排序尝试替代来源；不能为了让报告通过而无限抓取。

### 4.2 证据卡

中控为成功获取的页面生成证据卡。模型仍引用稳定 `W-*`，不自行抄写 URL。证据卡至少包含：

- 证据 ID、所属主题和原始查询；
- 规范 URL、发布者、作者、发布日期和获取时间；
- 来源等级、原始/转载关系和分级理由；
- 页面标题、正文短摘录及可定位位置，例如章节、页码或段落序号；
- 内容哈希、内容类型和获取状态；
- 该证据直接支持、反驳或限制的事实范围。

报告不应复制长篇原文；证据卡只保存核查所需的短摘录和定位信息。Researcher 看到的是清洗后的证据卡，不是任意网页脚本、导航、评论区或整页内容。

### 4.3 结论与证据对应

Researcher 的结构化产物应额外列出关键事实性结论及其 `W-*` 支持关系。中控至少确定性检查：

- 每个关键事实都有同主题证据；
- 数字、时间、主体和适用范围没有超出证据；
- S4 不能成为关键事实的唯一证据；
- 多源验证确实来自相互独立的发布链，而不是同一稿件的多个转载页；
- 发现来源冲突时，正文呈现冲突和不确定性，不能任意挑选一方。

关键数字、规则和高影响判断优先要求 S1/S2；找不到原始来源时，可由两个相互独立的 S2/S3 支撑并明确局限。探索性推断仍可保留，但必须与外部事实分开表达。

### 4.4 Reviewer 边界

Reviewer 只接收成稿实际使用的证据卡和必要摘录，不接收全部搜索噪声。它负责判断语义上的支持、遗漏、冲突、过度推断和不确定性表达；来源 ID、等级、URL 安全、主题归属和覆盖数量仍由代码门禁负责。

Reviewer 不应因为现实中不存在一手来源就机械否决主题，但必须确认正文已经披露证据限制。必要时可将研究审查配置为独立模型；不能假设“换一个 Agent 名称”就消除了同模型的相关性盲点。

## 5. 联网与不可信内容安全

获取原文会扩大网络攻击面，实施时必须先完成以下硬约束：

- 仅允许 HTTP(S)，禁止本地文件、内网、回环、链路本地地址和云元数据地址；DNS 解析和跳转后都要重新检查目标。
- 限制连接时间、重定向次数、响应大小和允许的内容类型；默认不执行 JavaScript、不提交表单、不携带登录 Cookie。
- HTML/PDF 解析与模型提示隔离。网页中的“忽略指令”“调用工具”等文字一律视为引用数据，不能改变任务、查询或工具权限。
- 去除脚本、样式、导航和重复模板，只保留可定位正文；解析失败不得把原始二进制或无限长度文本交给模型。
- URL、摘录和哈希保留审计，但 API 密钥、私人查询和本地路径不得写入证据卡或报告。

任何安全校验失败都应换用候选来源或舍弃该主题，不允许降级绕过网络边界。

## 6. 成本与失败边界

研究加深不应恢复开放式循环。建议保持：

- 先搜索和排序，后获取少量原文；只把与成稿有关的短证据交给模型。
- 搜索、页面获取和正文抽取分别缓存；内容哈希变化时才重新抽取。修订和 Reviewer 退回复用同一证据集。
- 每个主题的搜索、补充查询、页面数、总字节、模型上下文和墙钟时间都有硬上限。
- 单个主题证据不足时可以舍弃该主题并继续其他主题；所有主题都不足时终止研究阶段，保留旧报告。
- 不因获取失败自动改用搜索摘要支撑关键事实；允许摘要仅作为发现线索或明确标注的低风险背景。
- 缓存键继续包含输入、模型、搜索配置和流水线版本，旧版摘要证据不能误命中新版原文证据流程。

## 7. 评测与上线门槛

升级不能直接替换自动周报/月报。建议分四步：

1. **当前观察**：只统计现有流程，不改输出。
2. **离线回放**：用历史真实失败和成功案例验证排序、抓取、安全门禁及证据对应，不调用或少量调用模型。
3. **影子/手动试运行**：新流程生成候选结果，但不覆盖自动报告；与稳定基线做人审对比。
4. **有限上线**：先用于明确触发的手动报告，达到门槛后才进入自动报告；保留按流水线版本回退的能力。

首轮评测集建议包含至少 20 个有代表性的真实案例，覆盖零结果、来源冲突、PDF、转载、网页不可达、中文来源、时效性主题和 Reviewer 退回。对少量代表案例做多次模型运行，避免把随机的一次成功当作稳定能力。

上线前至少满足：

- 最终报告成功率达到 50%，且不低于稳定观察期基线；
- 人工抽检的关键事实证据支持率不低于 95%；
- 抽检中不受支持的高影响事实、S4 单独支撑关键事实、危险 URL 获取均为 0；
- 能找到 S1/S2 的主题中，正文实际使用权威来源的比例不低于 90%；
- 中位 token 增幅原则上不超过基线 30%，搜索、获取和最坏重试次数始终有界；
- 人工审阅确认研究深度确有提升，而不是只增加引用数量和正文长度。

这些比例是首次上线门槛，不是对模型永久质量的保证；观察数据表明门槛不现实或不足时，应先记录理由再调整，不能为了上线事后降低质量定义。

## 8. 数据和兼容性原则

当前观察期不需要删除数据库，也不需要修改数据库结构。未来第一阶段应优先把证据卡、分级和获取遥测保存在现有 `research_search` 产物的 `payload_json` 中，以减少结构变化；如果以后需要跨运行的全局正文缓存，再单独设计生命周期、容量、版权边界和数据库升级方案。

无论是否升级：

- 旧报告不回写、不批量重建；
- 已生成 Markdown 继续包含完整正文和实际外部链接，阅读不依赖 SQLite；
- 删除数据库仍会丢失运行审计、缓存、画像和反馈，不能作为常规升级步骤；
- 新旧流水线产物必须带不同版本，禁止跨不兼容证据格式复用缓存。

## 9. 启动升级的条件与实施顺序

只有稳定观察期结束、现有流程没有新的高优先级可靠性回归，并且人工抽查确认来源质量问题具有重复性时，才进入实现评审。推荐顺序为：

1. 建立基线统计和人工评审表；
2. 实现安全页面获取、正文抽取和证据卡，不改变报告生成；
3. 实现来源分级、转载识别和候选排序；
4. 增加关键结论与证据映射及确定性门禁；
5. 调整 Researcher/Reviewer 输入契约；
6. 完成离线、影子和手动试运行后再决定是否启用自动报告。

任一阶段如果显著降低成功率、产生不可控联网行为、让成本失去上限或无法证明质量提升，应停在稳定基线，不继续扩大改动。
