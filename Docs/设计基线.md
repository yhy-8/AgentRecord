# AgentRecord 设计基线

> 状态：架构基线，确认于 2026-07-15。本文是项目定位、数据边界和代码架构的唯一设计依据；README 只说明使用方法。

## 1. 项目定位

AgentRecord 是一个本地优先的个人记录与探索性分析系统。

用户记录的主体是日常事实、做过和计划做的事、观点、判断、问题与灵感。记录本身应当像普通日记一样低摩擦、可靠、可直接阅读。AI 的价值不在于随时回答常识问题或陪用户聊天，而在于跨越零散记录，发现值得继续思考的关联、变化、矛盾、盲点和研究方向，并将结果整理成独立报告。

核心价值链是：

```text
自然记录 → 可靠保存 → 引用与追溯 → 周期性整理 → 发散探索 → 独立报告
```

系统的目标不是替用户结束思考，而是提高下一轮思考的质量。

### 明确不做

- 不提供 `@AI` 即时聊天入口；知识问答、常识咨询和开放聊天交给现成 AI 产品。
- 不把日记变成结构化表单，不强制用户预先分类。
- 不让 AI 改写用户原文或替用户确认长期观点。
- 不把每条记录自动变成任务，也不为了展示 AI 能力制造弱关联。
- 不在需求形成前建设自由讨论式多 Agent、复杂 GUI 或专用向量数据库。

## 2. 产品原则

### 2.1 捕获优先

普通输入立即追加到当天 Markdown 日记。断网、模型失败或后台分析失败都不能影响原始记录保存。

“当天”和记录时间以用户最后按下回车、提交完整内容的时刻为准，而不是开始输入的时刻。一次提交只能捕获一个本地时间值，并用它同时决定日记文件名、文件头和记录行时间；跨午夜输入不得拆分，也不得因多次读取时钟形成日期不一致。

### 2.2 原文与派生结果分离

用户原话是最高优先级的事实源。总结、标签、关系、索引和报告都是派生结果，必须可追溯、可重建、可否决。AI 不能静默覆盖原始记录。

日记顶部 `<summary>` 是唯一允许程序更新的日记区域；更新必须通过 `AgentRecord/journal.py`，采用临时文件替换。分析报告保存在独立目录。

### 2.3 报告优先

AI 的主要交付物是日记总结、周报、月报以及未来的专题报告，不是聊天记录。手动日报用于需要当天下钻的场景。报告应关注少量有依据的强关联、思想演化、潜在矛盾和待判断问题，能够脱离终端会话独立阅读。

### 2.4 人保留判断权

AI 可以提出候选关联和反例，但必须区分“用户记录”“外部事实”和“AI 推断”。最终方向、价值判断和现实决策由用户完成。

### 2.5 简单且可追溯

能用 Markdown 明确表达的关系，不提前引入数据库。只有当跨记录关系、检索规模和反馈闭环证明有需要时，才增加可重建的派生索引。

## 3. 交互基线

终端明确分为两个模式。启动时进入记录模式，提示符固定为 `>>`，不展示模型名称。

记录模式：

```text
普通文字                   立即追加到今日日记
/v [日期]                  查看日记
/ref [类型] [筛选词]       选择并引用日记、日报、周报或月报
/d                         删除今日最后一条记录
/c                         清空终端显示
/h                         显示记录模式帮助
/mode                      切换到报告模式
```

报告模式：

```text
/s [日期]                  生成指定日记顶部总结
/a daily [日期]            手动生成分析日报
/a weekly [日期]           手动生成日期所在自然周的分析周报
/a monthly [日期]          手动生成日期所在自然月的分析月报
/m                         永久切换总结和报告统一使用的模型
/h                         显示报告模式帮助
/mode                      切换到记录模式
```

报告模式不接收普通记录；记录模式中的报告命令也不执行。模式隔离用于避免误触发模型或把报告操作误写入日记。`/m` 更新 `config.yaml` 的 `current_model`，该选择同时用于手动总结、手动报告和自动任务，直到用户再次切换。

手动报告通过当前交互进程内的单个后台工作线程生成。任务启动后界面立即回到记录模式，用户可以继续写入；后台线程只提交完成事件，不直接操作终端，输入循环负责显示完成或失败通知并恢复正在输入的内容。每个窗口同时最多运行一个手动报告，避免同一用户无意中并发消耗模型和竞争报告文件。

后台手动报告运行时，Ctrl+C、Ctrl+D 等由程序处理的软退出应被取消并显示提示。用户仍可通过操作系统强制关闭窗口；程序只明确提示“完成前不要关闭”，当前阶段不实现跨平台关窗拦截、任务转移或断点恢复。系统计划任务仍是独立进程，不依赖交互窗口存活。

`/ref` 支持 `diary`、`daily`、`weekly`、`monthly`，也接受对应中文名称。未给类型时由终端询问；系统展示最近 20 个匹配来源，选择后可以在同一条引用记录中补充由该来源引出的新想法。

输入以 `@` 开头时不再触发模型，它和其他普通文字一样被原样记录。这避免旧习惯导致内容丢失，同时彻底取消即时聊天语义。

## 4. 引用格式

引用是原始记录流中的一种用户记录，不是 AI 派生数据。标准格式为：

```markdown
**14:32 [引用]:** [自动分析周报 | 2026-07-06 至 2026-07-12](<../AnalysisReports/Weekly/2026-07-06_to_2026-07-12_auto.md>)

这里记录由该报告继续展开的新想法；可以留空。
```

格式语义：

- 当前日记文件名给出引用日期，行首 `HH:MM` 给出引用时间。
- `[引用]` 是稳定类型标签，便于人和后续分析程序识别。
- 链接文本同时写明来源类型和周期，即使链接失效仍可理解。
- 链接使用相对路径和 `/` 分隔符，兼容 Windows 与类 Unix 环境并便于整体迁移。
- 被引用的文件不会被修改；删除引用只删除当前日记中的这条记录。
- 可选关联记录与引用位于同一条记录内，避免失去上下文。
- 生成分析报告时，程序会读取本周期最多 10 个去重引用来源，每个最多提供 12000 字符；只允许读取日记和报告目录内的 Markdown，不递归追踪引用。

旧日记无需迁移。引用能力只增加新的标准记录，不改变既有文件头、`<summary>`、分隔线和普通记录格式。

## 5. 数据分层

```text
原始层
  Records/YYYY-MM-DD.md
  ├─ <summary>：程序生成的当日摘要
  └─ 原始记录流：普通记录与 [引用] 记录

派生交付层
  AnalysisReports/
  ├─ Daily/YYYY-MM-DD_manual.md
  ├─ Weekly/YYYY-MM-DD_to_YYYY-MM-DD_manual.md
  ├─ Weekly/YYYY-MM-DD_to_YYYY-MM-DD_auto.md
  ├─ Monthly/YYYY-MM_manual.md
  ├─ Monthly/YYYY-MM_auto.md
  └─ Topics/                       # 未来专题报告

分析知识层
  AnalysisReports/.analysis.sqlite3
  Record → Evidence → Theme → Hypothesis → Research → Insight → Report

运行诊断层
  Log/AgentRecord.log
  └─ AgentRecord.log.1、AgentRecord.log.2
```

### 原始层

- 每天一个 UTF-8 Markdown 文件。
- 用户内容只追加，除用户明确执行删除外不回退。
- AI 不能直接写原始记录流。
- 总结更新失败不得损坏已有文件。

### 派生交付层

- 手动报告和自动报告使用不同的确定文件名；同一周期各自最多一份。
- 再次手动生成同一周期报告必须先确认，确认后原子覆盖原手动报告；自动任务只原子覆盖对应自动报告。
- 报告记录生成方式、生成时间和输入日期范围。
- 模型或网络失败不改变原始日记。

### 分析知识层

分析知识层使用 SQLite，保存分析运行、Agent 中间产物、带版本的知识节点以及节点之间的关系。它位于报告目录下，不进入原始日记目录，也不是原始记录的替代品。

- 原始日记仍是用户记录的唯一事实源；SQLite 只保存来源定位、必要摘录和派生分析，不复制并接管原始记录。
- 节点至少区分 `evidence`、`theme`、`hypothesis`、`research` 和 `insight`；关系可以表达支持、反驳、归属、演化、分叉、合并和引用等语义。
- 节点和关系均有稳定 ID、状态、置信度、创建 Agent、版本及替代目标。修改采用新版本替代旧版本，不静默覆盖历史判断。
- 状态至少包括 `candidate`、`accepted`、`rejected` 和 `superseded`。只有通过 Reviewer 的内容才能进入最终报告上下文。
- 每次分析均保存独立运行记录和各 Agent 的结构化产物；失败运行可诊断，成功运行可继续迭代，同一周期再次分析可以读取先前已接受内容。
- 数据库写入使用事务、外键、忙等待和 WAL。未来架构迁移必须按 `PRAGMA user_version` 升级，升级前保留可恢复副本；数据库损坏时不得自动删除或覆盖。
- 从 Markdown 可以重建来源和 AI 派生节点，但用户未来对候选内容做出的接受、否决和修订属于额外判断，迁移和备份时必须保留。

### 运行诊断层

- 使用 Python 标准日志，记录运行 ID、阶段、Agent、状态和结构校验错误，不记录 API 密钥、完整日记、模型请求正文或模型响应正文。
- 日志目录由 `config.yaml` 的 `log_dir` 指定，默认是应用根目录下的 `./Log`；当前文件为 `AgentRecord.log`。单个文件达到 5 MiB 后按大小轮转，最多保留 2 个归档；产生新归档时自动替换最旧归档。
- 日志只用于定位运行故障，不是分析知识或用户记录。`Log/` 不进入版本库，也不能作为报告事实来源。

## 6. 代码架构

应用代码使用正式的 `AgentRecord` 包；仓库根目录只保留入口、配置、文档、测试和构建资料：

```text
main.py                         极薄的脚本与 PyInstaller 入口
AgentRecord/
  ├─ settings.py                模型选择和数据目录配置
  ├─ logging_config.py          标准日志与按大小轮转
  ├─ journal.py                 原始日记唯一读写边界
  ├─ ai_client.py               OpenAI 兼容协议与授权工具循环
  ├─ cli/
  │   ├─ entry.py               进程参数与 Windows 后台控制台处理
  │   ├─ terminal.py            跨平台输入和 Rich 展示
  │   ├─ commands.py            记录/报告模式命令映射
  │   ├─ report_jobs.py          单窗口手动报告后台任务状态
  │   └─ app.py                 交互主循环
  ├─ analysis/
  │   ├─ context.py             周期输入、引用和历史上下文
  │   ├─ orchestrator.py        总结与多 Agent 报告中控
  │   ├─ store.py               SQLite 运行、节点、关系和产物
  │   └─ automation.py          到期任务、锁和系统任务安装
  └─ agents/
      ├─ base.py                统一调用协议和权限提示；不接触持久化
      ├─ graph.py               节点与关系的共享结构校验
      ├─ extractor.py           原始记录 → Evidence
      ├─ cluster.py             Evidence → Theme 与时间轨迹
      ├─ explorer.py            Theme/Evidence → Hypothesis/Insight
      ├─ world.py               研究请求 → Research
      ├─ reviewer.py            候选决定与最终报告审查
      └─ report.py              已接受节点 → Markdown 报告正文
```

边界规则：

- 根目录 `main.py` 只导入并调用 `AgentRecord.cli.entry.main`，不包含业务逻辑。
- `AgentRecord.journal` 是原始日记的唯一写入边界；分析代码和 Agent 只能调用它公开的受控接口。
- `AgentRecord.ai_client` 只处理模型协议、只读日记工具和搜索，不决定何时生成何种报告。
- `analysis.orchestrator` 是唯一分析中控：准备输入快照、发配权限、调用 Agent、验证结构化输出、提交数据库事务，并在最终审查通过后原子保存报告。
- `analysis.store` 不读取日记、不调用模型、不决定分析语义，只保存中控已经验证的派生状态。
- `analysis.automation` 只调度中控公开接口，不包含 Agent 提示词或日记写入实现。
- `cli` 只映射交互和展示结果，不保存分析提示词、不直接写 Markdown。
- `cli.report_jobs` 只管理当前窗口的一条手动报告线程、状态和完成回调；它不实现分析流程。后台线程不能直接写终端，完成通知必须进入 `cli.terminal` 的线程安全队列。
- `agents/` 中每类 Agent 独立保存自己的职责提示、权限、输出契约和专属校验；共享调用及图结构校验分别位于 `base.py` 和 `graph.py`。这些模块无文件系统和数据库写权限，只接收中控提供的最小上下文并返回结构化候选结果。
- `AgentRecord.settings` 只负责配置。API 密钥不得进入日志、报告、测试输出或提交历史。

源码运行时 `config.yaml` 仍位于仓库根目录；PyInstaller 运行时仍位于 EXE 同目录。`diary_dir`、`analysis_dir` 和 `log_dir` 的相对路径均以该目录为基准。移动包位置不能改变配置或系统后台任务的解析基准。根入口、`python -m AgentRecord` 和打包 EXE 使用同一 `cli.entry`。

所有 Agent 初期统一使用 `current_model`，不增加按 Agent 选择模型的配置。职责差异由输入、输出格式和工具权限表达，而不是由不同模型制造。

## 7. 分析流程

```text
日常记录与显式引用
        │
        ▼
Markdown 原始记录（立即保存）
        │
        ├── 跨日总结 → 原日记 <summary>
        │
        └── 探索性分析
              ├─ 输入快照与来源编号
              ├─ Extractor：提取证据
              ├─ Cluster：主题和时间轨迹
              ├─ Explorer：关联、矛盾和候选洞见
              ├─ World：按需外部核查
              ├─ Reviewer：候选内容质量门禁
              ├─ Report：组织报告草稿
              └─ Reviewer：最终报告审查
                       │
                       ▼
                 原子保存报告
```

### 7.1 Agent 职责与权限

系统固定区分六类 Agent，另由不使用模型的中控统一调度。Agent 之间不自由对话，也不互相授予权限；所有信息必须经过中控以结构化数据传递。

1. **Extractor**：把原始记录和显式引用解析为事件、想法、判断、决定、问题和计划等证据节点。它能读取本次输入快照，不能联网。
2. **Cluster**：把证据组织成主题簇与时间轨迹，提出归属、延续、增强、衰退、分叉和合并关系。它只读取结构化证据及有限的已接受历史节点。
3. **Explorer**：寻找少量强关联、观点演化、潜在矛盾、盲点和替代解释，输出候选洞见及必要的研究请求。它不能自行把外部猜测写成事实。
4. **World**：只在 Explorer 明确提出研究需要时运行，通过被授权的搜索工具核查外部事实、时效性和反例。发送给第三方搜索的查询必须去除不必要的私人姓名、地址和原文细节。
5. **Reviewer**：检查来源、证据强度、来源身份、推断越界、弱关联和外部事实可靠性；决定候选节点接受、修改或拒绝。它还会在成文后再次审查报告。
6. **Report**：只使用通过 Reviewer 的节点组织最终 Markdown，不得在写作时新增未经审查的核心结论，也不能直接保存文件。

中控对每类 Agent 声明并执行：是否可读原始记录、可见的节点类型和状态、允许提出的节点与关系类型、允许调用的工具、是否可联网。默认拒绝未声明能力。任何 Agent 都不能直接修改日记、SQLite、自动任务状态或最终报告。

### 7.2 结构化产物与迭代

- Agent 输出必须是可验证的 JSON，而不是供下一个 Agent 猜测语义的自由文本。
- 每个证据节点携带日记相对路径、日期、时间和记录序号；同一分钟多条记录不能共用一个来源 ID。
- 下游 Agent 可以在 `source_refs` 中引用中控提供的可见知识节点 ID；中控必须确定性地展开这些节点的原始来源并去重，不能让 Agent 自行复制整条来源链。无法展开的引用仍视为越权并拒绝。
- Agent 输出通过 JSON 解析但未通过权限或结构校验时，中控也要保存一份 `failed` 中间产物和明确错误，便于定位失败阶段；日志只记录安全的结构信息，不复制该产物正文。
- 候选洞见必须列出支持证据、可能反证、推断等级和研究需要。没有足够依据的内容保持候选或被拒绝，不能进入报告。
- 同一周期再次运行时，中控可向 Agent 提供该周期上一成功运行以及近期其他周期的已接受节点。Agent 可以复用、补充或提出替代版本，不能原地篡改旧版本。
- Reviewer 接受替代版本后，旧版本转为 `superseded`；新版本被拒绝时旧版本保持有效。所有变化保留审计链。
- 一次运行中的修改循环有明确上限：候选审查最多修订一次，报告审查最多修订一次。超过上限视为本次生成失败，保留诊断信息但不覆盖已有报告。

### 7.3 报告整合和质量门禁

当前日报、周报和月报均经过 Extractor、Cluster、Explorer、Reviewer 和 Report；World 仅在 Explorer 提出有效研究请求时运行。月报先分块提取，再聚合结构化证据，不能把整月无限增长的原文一次交给最终模型。

最终报告只选择少量通过审查的高价值内容。主要洞见应说明发生了什么、来源依据、可能解释、反证或其他解释、为何值得关注以及留给用户判断的问题。无材料的章节不为满足模板而生成。

最终 Reviewer 和确定性校验共同保证：

- 关键判断都能追溯到实际来源或已保存的外部研究结果。
- 用户记录、历史报告、AI 推断和外部事实身份不混淆。
- 时间先后不被无依据地写成因果，弱关联不被写成确定结论。
- 报告不逐条复述、不制造任务、不包含无法解析的来源 ID。
- 外部事实带来源和核查时间；无法核实时明确保留不确定性。
- 审查失败、模型失败或数据库失败时，不覆盖已有报告，不改变原始日记。

### 7.4 周期闭合与系统后台任务

周期只按运行机器的本地日历计算：自然日为 00:00–23:59，自然周为周一至周日，自然月为每月 1 日至最后一日。未闭合的周期绝不自动生成，以免分析仍在变化的输入。

自动任务必须独立于交互程序运行：

1. 用户显式安装系统后台任务；Windows 使用当前用户的任务计划程序，类 Unix 使用当前用户的 cron。
2. Windows 在当前用户登录时、类 Unix 在系统重启时立即启动一次检查；两者此后均在每小时第 5 分钟检查。
3. 每次检查使用独立进程，执行完成后退出；关闭终端或 AgentRecord 不影响后续检查，也不需要维持常驻 Python 进程。
4. 日总结最早在次日 00:00 后生成，写回前一日日记的 `<summary>`。
5. 周报最早在下周一 00:00 后生成，覆盖上一完整周一至周日的自动报告。
6. 月报最早在下月 1 日 00:00 后生成，覆盖上一完整自然月的自动报告。

类 Unix 系统任务通过 `python main.py --install-automation` 安装，Windows 打包版本通过 `AgentRecord.exe --install-automation` 安装，并可用对应的 `--uninstall-automation` 卸载。任务实际调用同一入口的 `--run-automation`；Windows EXE 在该参数下隐藏控制台窗口，正常交互入口不受影响。自动检查使用跨进程锁，避免慢任务重叠写入。

默认自动流程是“日总结 + 周报 + 月报”。日报只保留手动下钻，因为日总结已经承担每日压缩，再自动生成日报会增加重复产物和模型消耗。

调度按日总结、周报、月报的顺序执行，但三者保存独立成功位置、失败信息和重试进度。单项失败不会回退已有产物，也不会阻断其他可以直接基于原始记录运行的周期任务。失败任务在下一小时检查时重试。

首次启用只处理各自最近一个闭合周期，避免对全部历史记录产生意外成本；形成状态后，从上次成功位置连续补做程序关闭期间遗漏的周期。无记录周期只推进检查位置，不生成空文件。状态保存在 `AnalysisReports/.automation-state.json`。

固定路径保证幂等：相同周期的自动报告不会因重复检查产生副本；手动报告与自动报告互不覆盖。

手动报告与系统自动任务是两种不同的后台机制：手动报告属于当前窗口内的工作线程，窗口存活时在完成后通知用户；系统自动任务属于操作系统启动的独立进程，使用 `.automation.lock` 防止两个调度进程重叠。两者都只在短事务中写 SQLite，等待模型期间不持有数据库事务，也不阻塞前台 Markdown 追加。

### 日报关注

- 当天重要的新想法、变化和注意力中心。
- 引用所形成的明确延伸关系。
- 与历史记录少量、可追溯的强关联。
- 潜在矛盾、盲点和少量值得继续判断的问题。

### 周报关注

- 本周主要思想簇及其变化。
- 新出现、持续强化或衰退的主题。
- 反复出现但未解决的问题。
- 观点之间的矛盾、转变和可能原因。
- 下一周期最值得继续探索的少量方向。

### 月报关注

- 一个月内注意力和精力如何分配，哪些领域实际占据了持续关注。
- 长期主题的出现、增强、衰退、分叉或合并。
- 重要判断在后续记录或外部事实中得到的支持、挑战和修正。
- 跨周反复出现的行为、问题、限制条件和潜在盲点。
- 下个月最值得继续探索的少量方向，而不是完整任务清单。

月报综合当月原始记录、显式引用、月前 30 天总结、与该月相交的已有周报以及 SQLite 中仍有效的历史节点。下级报告只作为 AI 分析材料，不能被表述成用户已经认可的观点。

## 8. 演进路线

SQLite 运行记录、版本化节点关系、六类 Agent 流水线、来源继承、周期重跑、系统自动任务和手动报告后台线程已经落地。下一阶段优先观察真实报告质量和运行稳定性，再决定是否实现：

1. 用户在终端接受、否决或修正候选关联的交互。
2. 手动报告任务的跨进程持久化、取消、恢复和全局去重。
3. 基于真实检索瓶颈增加关系查询、FTS5 或专题报告。
4. 只有普通索引无法满足规模和语义检索时，才评估向量能力。

每一步都必须保持：旧日记可无损读取、派生数据可重建、AI 失败不影响记录、报告引用能够追溯到来源。
