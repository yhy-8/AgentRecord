# 各 Agent 功能与联动

本文描述每日人物画像、每日信息收集以及周报、月报中各个 AI Agent/专用角色的功能、输入边界和联动关系。报告与人物画像中控位于 `analysis.orchestrator`，四个正式 `AgentSpec` 位于 `agents/`；每日信息的两个专用 AI 角色由 `analysis.information` 编排。所有角色都不能直接写日记、报告或数据库。

## 1. 总体流程

```text
周期记录 + 日记引用 + 历史画像 + 每日信息简报
          │
          ├─ Retrospective → 整理与回顾 + 画像候选
          │                         ↓
          │                    Reviewer 审查
          │
          └─ ResearchPlanner → 1~5 个公开研究主题
                                    ↓
                         中控固定查询一次 + W-* 证据
                                    ↓
                         Researcher 基于证据生成正文
                                    ↓
                               Reviewer 审查
                                    ↓
                    中控组装、来源索引、原子交付
```

第一板块由 Retrospective 生成，第二板块由 Researcher 生成。ResearchPlanner 只选择领域与去隐私查询，Reviewer 只做质量门禁。所有 Agent 使用同一 `current_model`；职责差异来自输入、提示、工具权限和输出契约。

## 2. 每日信息收集 Agent

每日信息链路先于周报、月报运行，输出 `AnalysisReports/Information/YYYY-MM-DD.md`。它包含两个职责隔离的专用 AI 角色：

1. **DailyQueryPlanner（定向选题）**：只读取本周截至当天的用户原始记录，不读取旧简报、旧简报正文、“可继续追踪”或历史查询；它不联网，提出零至三个由本周记录直接引出的公开搜索问题。每个查询必须返回真实的本周 `R-*` 记录依据，中控拒绝无记录依据或引用未知记录的查询。程序随后清洗邮箱、长数字和本地路径，并使用自己保存的历史查询指纹确定性去重；历史查询不会进入 Planner 上下文，因此不能反向产生选题。没有新角度或 Planner 连续两次不能给出合法结构时，定向部分降级为空，综合新闻仍可继续。
2. **DailyInformationCollector（信息判断）**：中控确定性执行两条综合新闻查询和零至三条去隐私定向查询，为每条合法搜索结果分配 `I-Q001-001` 形式的证据 ID。Collector 只看到每个 ID 的标题、域名、日期和短摘要，看不到 URL、搜索服务原始排版或旧简报正文，也没有工具权限。它返回结构化 JSON，不负责 Markdown、链接、`T-*` 或 `R-*` 排版。

“今日值得关注”允许零至五项，数量由当日证据质量决定，不得凑数。每项分别返回具体变化、可核查细节、关注理由和证据 ID；日程预告、报纸出刊、宽泛评价或“市场关注”本身不能成项。“与本周思考相关的探索”对每个仍有搜索结果的选题返回 `supported` 或 `insufficient_evidence`；泛泛或不相关的结果必须标为证据不足，只有 `supported` 项才进入简报。“可继续追踪”保存未来真正值得等待的具体问题，但不进入以后 Planner 的输入。

旧简报只被中控解析成紧凑去重索引：日期、已覆盖标题和实际使用的来源 URL。`可继续追踪` 正文不会进入索引；Collector 只看到其中的日期、类型和标题，中控自己使用 URL 判断本轮来源是否已经出现。若再次使用本周已出现的来源，Collector 必须说明实质新增内容。中控最终校验证据 ID 的查询归属，并确定性渲染 Markdown、外部 URL、主题编号和本周记录依据，模型无法抄错或编造链接。

每份简报附带两个不可见 HTML 注释：`agentrecord-targeted-queries` 保存当天提出的定向查询供程序去重，`agentrecord-information-index` 保存实际入选标题和来源供后续简报查重；两者都不包含“可继续追踪”正文。当天重跑只读取当天以前的同周索引，跨自然周不继承查询禁用清单。

正常路径固定调用 Planner 和 Collector 各一次。整个每日任务共享一次结构修订额度，因此最多三次模型调用：Planner 使用修订后仍不合法时降级为空；Collector 没有剩余修订额度或唯一修订仍失败时终止。部分搜索失败但仍有合法证据时使用成功结果继续；全部无证据才终止。非网络内容失败首次即暂停自动重跑，网络和限流仍按 5 分钟策略处理，用户可明确执行 `/retry`。同输入失败运行中已完成的 Planner、搜索证据或 Collector 结构可从 SQLite 复用，避免重新调用和搜索。

每日信息简报不是日报，也不直接更新人物画像。它在周报、月报中只作为 ResearchPlanner 的选题线索，Researcher 必须重新联网核查，不能把简报文字直接当成已确认事实。

自然周边界是硬边界：周一只读取周一截至生成时已有的记录，不依赖周日总结，也不会读取上一周记录。默认早晨生成时若周一尚无记录，零个定向选题是正确结果；综合新闻仍按实际证据生成零至五项。

```text
本周记录 ────────→ DailyQueryPlanner → 带 R-* 依据的查询（可为 0 项）
                                  ↓
历史查询指纹 ─────→ 中控确定性去重、去隐私
                                  ├─ 两条综合新闻查询
                                  └─ 零至三条定向查询
                                           ↓
                                  I-* 标准化证据
                                           ↓
旧简报紧凑索引 ─────────→ DailyInformationCollector JSON
                                           ↓
                          中控校验 ID、渲染链接与 R-*
                                           ↓
                                      每日信息简报
```

### 2.1 每日人物画像

每日画像任务在日总结之后、信息简报之前处理前一自然日的原始记录。它复用 Retrospective 与 Reviewer：Retrospective 生成一份仅供审查的内部摘要和零至若干画像候选，Reviewer 逐项核对来源并决定接受或拒绝。内部摘要不写成日报；新闻、一次性事件、任务和 AI 建议不能进入画像。

候选先关联 `running` 的 `daily_profile` 运行，只有整个审查完成并把运行标记为 `completed` 后才进入历史画像和替代旧版本。某日没有稳定变化时以零候选完成，SQLite 中的 completed 运行仍证明当天已检查。失败运行的候选继续保留审计但不可见。

## 3. 运行与输入快照

报告入口只接受 `weekly`、`monthly`；每日画像使用独立的 `daily_profile` 运行，每日信息简报使用 `daily_information` 运行。三类工作流都会把实际使用的标准日记记录登记为稳定来源 ID：

```text
R-YYYYMMDD-NNN-HHHHHHHHHHHH
```

末尾 12 位十六进制哈希由日期、时间、标签、说话者和完整正文共同计算。同一位置内容未变时 ID 稳定；删除后重写或编辑同序号内容会得到新 ID，不会复用旧证据身份。旧数据库和旧报告中的 `R-YYYYMMDD-NNN` 仍可读取与校验。来源目录保存相对文件、日期、时间、记录序号、说话者、标签、内容哈希和最多 500 字符摘录；若同一 ID 意外对应不同正文，中控拒绝覆写而不是改写历史证据。

运行快照包含：

- 本周期原始记录；
- 本周期记录中明确引用的日记，最多十份且每份最多解析前 12000 字符；被引用日记的每条记录同样获得 `R-*` 身份并进入来源目录，不再作为无法引用的原始大段文本；
- 周期开始前 30 天的日记总结；
- 完整落在该自然月内的同期周报，作为 AI 已有整理而非用户原始观点；跨月周不读取，同周手动/自动版同时存在时优先手动版；
- 周期内每日信息简报，作为待重新核查线索；
- 截止周期结束日仍有效的人物画像。

完整快照用于输入哈希和审计；模型输入另有确定性预算。Retrospective 总输入超过 120000 字符时，原始记录按约 30000 字符分块，每块分别经过生成、校验和 Reviewer，再合并结果；单条超长记录也会保留同一 `R-*` 身份分片处理。历史画像按最近观察时间选入最多 24000 字符，并显式提供省略数量，未选入条目仍保留在数据库。若 ResearchPlanner 的原始记录输入仍超限，它只从已过审且带 `R-*` 引用的 Retrospective 与信息简报中选题。

报告引用目标不会被载入。月报内部读取同期周报不等同于用户引用：它只提供二级整理上下文，最终事实判断仍必须回到 `R-*` 来源。

快照计算 SHA-256 并写入 `analysis_runs`。中控先创建 `running` 运行，再进行任何 Agent 工作。

## 4. 历史画像隔离

历史画像按报告截止日重建：候选所属运行的 `period_end` 和条目 `last_observed` 都不得晚于截止日，以后的用户修正/否决与 Retrospective 替代也会按它们的生效日期回放，不会泄漏到过去报告。

SQLite 的 32 位 ID 不直接交给模型。中控在一次调用内将画像映射为 `P001`、`P002` 等短别名，模型输出后再映射回持久 ID。Retrospective 只能替代本次可见画像。

## 5. Retrospective：整理与回顾

Retrospective 可读取原始记录、日记引用、近期总结、月报辅助周报和历史画像，但不能联网。它同时返回：

1. 第一板块 Markdown；
2. 最多 12 条画像候选。

正文必须回顾事实、行动、决定与问题，并整理观点、理念、理想、关注领域和行为模式的变化。每个内容段落必须就近包含输入中实际存在的 `[R-YYYYMMDD-NNN-HHHHHHHHHHHH]`（旧来源可以没有哈希后缀），不得把时间先后写成因果，不得心理诊断或给出行为教练式命令。

画像候选只允许：

- `viewpoint`
- `principle`
- `ideal`
- `behavior_pattern`
- `interest`

一次性事件、任务、新闻事实和 AI 建议不能进入画像。每个候选至少有一条本周期记录来源；`behavior_pattern` 还必须由至少两条不同记录支持。历史来源可以补充连续性，但不能单独支撑“本期发生更新”。同一输出中的重复候选会被拒绝；与当前可见画像完全相同的候选也不能伪装成新条目。

每个 Agent 在一个报告阶段最多调用三次。JSON 解析、结构或内容校验失败时，中控把未通过输出和确定性错误交回同一 Agent 定向修订；Reviewer 否决时，中控把原稿与完整审核意见交回原生成 Agent，再执行全部校验和复审。三次仍失败才终止整次报告，已有报告保持不变。修订后缀位于不变的原始请求之后，以提高提示缓存复用机会。

## 6. Reviewer：第一板块门禁

Reviewer 不联网。中控只给它第一板块实际引用的最小记录集合、相关历史画像和板块候选，使它能比较 `[R-*]` 是否真的支持相邻判断，而不只是检查引用格式。它检查：

- 事实与时期是否混淆；
- 是否把引用或 AI 内容当成用户立场；
- 来源是否覆盖核心判断；
- 是否出现因果越界、心理诊断、套话或行为教练；
- 每条画像候选是否值得跨周期保留。

Reviewer 必须对每个候选且仅对每个候选返回 `accepted` 或 `rejected`。画像候选被拒绝本身不代表正文失败；只有正文存在无依据判断或其他实质质量问题时才令板块不通过。缺项、重复、未知 ID 或否决时没有具体意见会进入有限审查格式修订。

第一板块通过后，中控根据来源目录计算 `first_observed`、`last_observed`。被接受的新版本才会把旧画像改为 `superseded`；被拒绝的候选保留审计记录，但不进入以后上下文。

## 7. ResearchPlanner：研究选题

ResearchPlanner 读取本周期记录、已经通过的第一板块和每日信息简报，根据材料和研究价值自主选择一至五个公开领域问题，不为凑数拆分或补写。主题来源标为：

- `records`：由记录引出，必须带 `R-*`；
- `news`：由综合信息雷达引出，可以没有日记来源；
- `mixed`：两者共同引出，必须带 `R-*`。

它不能联网，也不生成报告正文。查询在确定性校验前再次清洗邮箱、七位以上长数字、Windows/Unix 本地路径等私人信息。研究题目必须能通过公开资料实质研究，不能把私人细节送入搜索。

## 8. 中控检索与 Researcher：领域探索与研究

使用第三方搜索时，联网是中控的确定性步骤，不是 Researcher 的自由工具循环。中控逐字执行 ResearchPlanner 的每个固定查询一次，每个查询最多接受 10 条结果，过滤重复或非法 URL，并为证据分配：

```text
W-Q001-001
```

完整查询、URL 和检索遥测保存到 `research_search` 审计产物。Researcher 只能看到经过清洗的主题，以及证据的 `W-*`、标题、摘要和日期；它不能读取完整原始日记、看不到 URL，也没有工具权限。网页标题和摘要仍被视为不可信数据，不能覆盖程序指令。

第二板块需要：

- 按检索后保留主题的顺序，为每个主题输出一个与 Planner `title` 完全一致的三级标题；
- 每个标题下只放置该主题的记录来源、外部证据、分析和推演；
- 优先一手、权威、可核查资料；
- 同时寻找支持、反例、边界、相邻概念和不同视角；
- 记录驱动主题保留相应 `R-*`；
- 外部事实就近引用准确的 `W-*`；
- 将探索性推断明确标为推断或假设；
- 不输出新闻链接堆砌或行动指令。

中控检查三级标题与保留主题的数量、文字和顺序完全一致，每个主题块的记录引用和外部证据也必须属于该主题。每个 `W-*` 必须存在且放在对应选题下；模型输出任何 HTTP(S) URL 都会被拒绝。通过后，中控才把 `W-*` 替换为搜索服务返回的精确 Markdown 链接并生成 `sources`，因此不存在模型抄错、补全或猜测 URL 的协议失败。某个查询零结果时可丢弃该主题，最终标题数量随实际保留主题而变；全部查询零结果才终止研究阶段。

每日信息简报和报告入口必须配置第三方搜索，以便中控审计每条固定查询与证据。模型原生搜索只保留给非严格的通用调用，不会作为简报或报告的降级路径。

## 9. Reviewer：第二板块门禁

Reviewer 单独检查领域研究：

- 来源是否支持正文中的事实；
- 是否只挑支持材料而没有反例或适用边界；
- 记录与研究问题之间是否被夸大；
- 探索性推断是否清楚标注不确定性；
- 是否越权替用户做最终判断。

Researcher 的格式/证据校验或 Reviewer 内容审查失败时，由中控把原稿与原因交回 Researcher 有限修订、校验和复审；第三方搜索证据在同一运行的修订间复用，不会重复付费搜索。最多三次仍不通过才终止，已有同路径报告保持不变。

## 10. 组装与交付

中控确定性组装：

```markdown
## 一、整理与回顾

...

## 二、领域探索与研究

...

## 来源索引

- [R-...] 日期时间 — 文件与记录序号
```

报告头写入生成时间、手动/自动来源、`manual|scheduled|retry` 触发方式、原始日记范围和运行 ID。

文件先写 `.tmp` 再原子替换固定路径。只有文件交付成功后运行才能标为 `completed`；完成状态写库失败时中控恢复旧文件或删除新文件，避免出现“文件成功但数据库失败”的分裂状态。

## 11. 失败与重跑

Agent 调用、解析、结构校验和审查产物均写入 `agent_artifacts`。若回答不是合法顶层 JSON，中控保存原回答、错误和已有工具遥测，再携带这些信息请求同一 Agent 修订。整个回答只是合法 JSON 外包一层 ` ```json ` 围栏，或完整 JSON 后只有兼容接口误加的孤立引号、结束括号或围栏字符时，可确定性恢复；带解释前缀、两个拼接对象或内部结构损坏仍进入修订，不猜测和补写内容。周/月报告的结构、内容和 Reviewer 门禁仍按各阶段最多三次执行；每日信息使用前述任务级一次修订预算。普通 API 失败不做内容修订。

每个审计产物保留耗时、HTTP 尝试数、token 用量、缓存命中/未命中 token 和联网证据。每日检索保存为 `daily_information_search`，报告检索保存为 `research_search`；查询、输入、模型和流水线版本一致时，失败运行重试可以复用已通过确定性校验的阶段。未过审稿件、空证据搜索、输入快照改变、模型改变或不兼容的流水线版本改变时均不命中。

同周期报告可以重跑并覆盖固定 Markdown 文件。每次运行在 SQLite 中仍有独立 ID、输入哈希、来源集合和触发方式。Markdown 报告不是稳定引用对象，运行记录仅用于审计和排障。
