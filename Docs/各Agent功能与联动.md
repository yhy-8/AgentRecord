# 各 Agent 功能与联动

本文描述每日人物画像、每日信息收集以及周报、月报中各个 AI Agent/专用角色的功能、输入边界和联动关系。报告与人物画像中控位于 `analysis.orchestrator`，四个正式 `AgentSpec` 位于 `agents/`；每日信息的两个专用 AI 角色由 `analysis.information` 编排。所有角色都不能直接写日记、报告或数据库。

## 1. 总体流程

```text
周期记录 + 日记引用 + 历史画像 + 每日信息简报
          │
          ├─ Retrospective → 整理与回顾 + 画像候选
          │                         ↓
          │                    Reviewer 审查
          │
          └─ ResearchPlanner → 1~3 个公开研究主题
                                    ↓
                         中控固定查询一次 + W-* 证据
                                    ↓
                         Researcher 基于证据生成正文
                                    ↓
                               Reviewer 审查
                                    ↓
                    中控组装、来源索引、原子交付
```

第一板块由 Retrospective 生成，第二板块由 Researcher 生成。ResearchPlanner 只选择领域与去隐私查询，Reviewer 只做质量门禁。所有 Agent 使用同一 `current_model`；职责差异来自输入、提示、工具权限和输出契约。

## 2. 每日信息收集 Agent

每日信息链路先于周报、月报运行，输出 `AnalysisReports/Information/YYYY-MM-DD.md`。它包含两个职责隔离的专用 AI 角色：

1. **DailyQueryPlanner（定向选题）**：读取本周截至当天的用户记录、本周此前的信息简报及其中保存的定向查询，不联网，提出零至三个适合公开搜索的新问题。长期兴趣不会因为再次出现在记录里就被当作新选题；只有新的事件、数据、争议、来源或待核查问题才允许继续追踪同一领域。程序还会清洗邮箱、长数字和本地路径，并确定性剔除与本周旧查询相同或高度相似的查询；没有新角度时可以不做定向搜索。
2. **DailyInformationCollector（联网收集）**：接收两条固定综合新闻查询、去隐私定向查询和本周此前的信息简报，必须实际使用 `web_search`。其中“今日值得关注”固定选择五项彼此独立的信息，用于纯粹拓宽视野；“与本周思考相关的探索”必须按 `T001` 至 `T003` 逐项对应 DailyQueryPlanner 的记录驱动选题；“可继续追踪”保存尚待观察的变化。第三方搜索遥测还必须证明每条授权查询都实际执行。每项事实都要提供链接。旧简报只用于查重，已报道的事件、背景和结论不得复述；同一主题只有出现实质更新时才能再次写入，并须说明新增内容。

每份简报会附带一行 Markdown 阅读器不显示的 HTML 注释，例如 `<!-- agentrecord-targeted-queries: [...] -->`。其中只保存当天零至三个定向查询及选题理由，程序在本周后续日期读取它做确定性查重；它不是报告引用，也不包含五项综合新闻正文。当天任务重跑时只读取当天以前的同周简报，不把即将覆盖的旧文件当作历史。跨自然周不继承查询禁用清单，避免长期兴趣被永久封锁，但周报、月报仍会综合周期内所有简报。

DailyQueryPlanner 的 JSON 或 DailyInformationCollector 的章节、固定五项、来源链接与联网遥测未通过确定性校验时，中控同样把未通过输出和具体原因追加到原请求末尾，最多共尝试三次。API 或网络失败不进入内容修订循环，仍由自动任务层按失败类别调度。

每日信息简报不是日报，也不直接更新人物画像。它在周报、月报中只作为 ResearchPlanner 的选题线索，Researcher 必须重新联网核查，不能把简报文字直接当成已确认事实。

自然周边界是硬边界：周一只读取周一截至生成时已有的记录，不依赖周日总结，也不会读取上一周记录。默认早晨生成时若周一尚无记录，零个定向选题是正确结果；综合新闻五项仍正常生成。

```text
本周记录 ───────────────┐
本周此前简报 + 查询历史 ─┴→ DailyQueryPlanner → 新的去隐私查询（可为 0 项）
                                                   ├─ 综合新闻固定查询
本周此前简报（查重）───────────────────────────────┤
                                                   ↓
                                         DailyInformationCollector
                                                   ↓
                                              每日信息简报
                                                   ↓
                                    周/月 ResearchPlanner 选题
```

### 2.1 每日人物画像

每日画像任务在日总结之后、信息简报之前处理前一自然日的原始记录。它复用 Retrospective 与 Reviewer：Retrospective 生成一份仅供审查的内部摘要和零至若干画像候选，Reviewer 逐项核对来源并决定接受或拒绝。内部摘要不写成日报；新闻、一次性事件、任务和 AI 建议不能进入画像。

候选先关联 `running` 的 `daily_profile` 运行，只有整个审查完成并把运行标记为 `completed` 后才进入历史画像和替代旧版本。某日没有稳定变化时以零候选完成，SQLite 中的 completed 运行仍证明当天已检查。失败运行的候选继续保留审计但不可见。

## 3. 运行与输入快照

报告入口只接受 `weekly`、`monthly`；每日画像使用独立的 `daily_profile` 运行。两者都会解析范围内的标准日记记录，每条记录得到稳定来源 ID：

```text
R-YYYYMMDD-NNN
```

来源目录保存相对文件、日期、时间、记录序号、说话者、标签、内容哈希和最多 500 字符摘录。

运行快照包含：

- 本周期原始记录；
- 本周期记录中明确引用的日记，最多十份且每份最多 12000 字符；
- 周期开始前 30 天的日记总结；
- 完整落在该自然月内的同期周报，作为 AI 已有整理而非用户原始观点；跨月周不读取，同周手动/自动版同时存在时优先手动版；
- 周期内每日信息简报，作为待重新核查线索；
- 截止周期结束日仍有效的人物画像。

报告引用目标不会被载入。月报内部读取同期周报不等同于用户引用：它只提供二级整理上下文，最终事实判断仍必须回到 `R-*` 来源。

快照计算 SHA-256 并写入 `analysis_runs`。中控先创建 `running` 运行，再进行任何 Agent 工作。

## 4. 历史画像隔离

历史画像只读取 `status=accepted` 且 `last_observed <= period_end` 的条目，防止以后发生的变化泄漏到过去报告。

SQLite 的 32 位 ID 不直接交给模型。中控在一次调用内将画像映射为 `P001`、`P002` 等短别名，模型输出后再映射回持久 ID。Retrospective 只能替代本次可见画像。

## 5. Retrospective：整理与回顾

Retrospective 可读取原始记录、日记引用、近期总结、月报辅助周报和历史画像，但不能联网。它同时返回：

1. 第一板块 Markdown；
2. 最多 12 条画像候选。

正文必须回顾事实、行动、决定与问题，并整理观点、理念、理想、关注领域和行为模式的变化。每个内容段落必须就近包含合法 `[R-YYYYMMDD-NNN]`，不得把时间先后写成因果，不得心理诊断或给出行为教练式命令。

画像候选只允许：

- `viewpoint`
- `principle`
- `ideal`
- `behavior_pattern`
- `interest`

一次性事件、任务、新闻事实和 AI 建议不能进入画像。每个候选至少有一条本周期记录来源；历史来源可以补充连续性，但不能单独支撑“本期发生更新”。

每个 Agent 在一个报告阶段最多调用三次。JSON 解析、结构或内容校验失败时，中控把未通过输出和确定性错误交回同一 Agent 定向修订；Reviewer 否决时，中控把原稿与完整审核意见交回原生成 Agent，再执行全部校验和复审。三次仍失败才终止整次报告，已有报告保持不变。修订后缀位于不变的原始请求之后，以提高提示缓存复用机会。

## 6. Reviewer：第一板块门禁

Reviewer 不联网。中控只给它第一板块实际引用的最小记录集合、相关历史画像和板块候选，使它能比较 `[R-*]` 是否真的支持相邻判断，而不只是检查引用格式。它检查：

- 事实与时期是否混淆；
- 是否把引用或 AI 内容当成用户立场；
- 来源是否覆盖核心判断；
- 是否出现因果越界、心理诊断、套话或行为教练；
- 每条画像候选是否值得跨周期保留。

Reviewer 必须对每个候选且仅对每个候选返回 `accepted` 或 `rejected`。画像候选被拒绝本身不代表正文失败；只有正文存在无依据判断或其他实质质量问题时才令板块不通过。缺项、重复、未知 ID 或否决时没有具体意见会进入有限审查格式修订。

第一板块通过后，中控根据来源目录计算 `first_observed`、`last_observed`。被接受的新版本才会把旧画像改为 `superseded`；被拒绝的候选保留审计记录，但不进入以后上下文。

## 7. ResearchPlanner：研究选题

ResearchPlanner 读取本周期记录、已经通过的第一板块和每日信息简报，选择一至三个公开领域问题。主题来源标为：

- `records`：由记录引出，必须带 `R-*`；
- `news`：由综合信息雷达引出，可以没有日记来源；
- `mixed`：两者共同引出，必须带 `R-*`。

它不能联网，也不生成报告正文。查询在确定性校验前再次清洗邮箱、七位以上长数字、Windows/Unix 本地路径等私人信息。研究题目必须能通过公开资料实质研究，不能把私人细节送入搜索。

## 8. 中控检索与 Researcher：领域探索与研究

使用第三方搜索时，联网是中控的确定性步骤，不是 Researcher 的自由工具循环。中控逐字执行 ResearchPlanner 的每个固定查询一次，每个查询最多接受 10 条结果，过滤重复或非法 URL，并为证据分配：

```text
W-Q001-001
```

完整查询、URL 和检索遥测保存到 `research_search` 审计产物。Researcher 只能看到经过清洗的主题，以及证据的 `W-*`、标题、摘要和日期；它不能读取完整原始日记、看不到 URL，也没有工具权限。网页标题和摘要仍被视为不可信数据，不能覆盖程序指令。

第二板块需要：

- 优先一手、权威、可核查资料；
- 同时寻找支持、反例、边界、相邻概念和不同视角；
- 记录驱动主题保留相应 `R-*`；
- 外部事实就近引用准确的 `W-*`；
- 将探索性推断明确标为推断或假设；
- 不输出新闻链接堆砌或行动指令。

中控检查每个 `W-*` 都存在、属于对应选题，并保证每个保留主题至少有一项实际引用；模型输出任何 HTTP(S) URL 都会被拒绝。通过后，中控才把 `W-*` 替换为搜索服务返回的精确 Markdown 链接并生成 `sources`，因此不存在模型抄错、补全或猜测 URL 的协议失败。某个查询零结果时可丢弃该主题；全部查询零结果才终止研究阶段。

模型原生搜索保留兼容路径：Researcher 可使用原生 `web_search`，但查询受中控白名单约束，同次重复查询不执行，正文链接和 `sources` 仍必须存在于实际搜索证据。这个兼容路径不改变第三方搜索的默认确定性工作流。

## 9. Reviewer：第二板块门禁

Reviewer 单独检查领域研究：

- 来源是否支持正文中的事实；
- 是否只挑支持材料而没有反例或适用边界；
- 记录与研究问题之间是否被夸大；
- 探索性推断是否清楚标注不确定性；
- 是否越权替用户做最终判断。

Researcher 的格式/证据校验或 Reviewer 内容审查失败时，由中控把原稿与原因交回 Researcher 有限修订、校验和复审；第三方搜索证据在同一运行的修订间复用，不会重复付费搜索。最多三次仍不通过才终止，已有同路径报告保持不变。原生搜索兼容路径的修订仍按服务商能力重新搜索。

## 10. 组装与交付

中控确定性组装：

```markdown
## 一、整理与回顾

...

## 二、领域探索与研究

...

## 来源索引

- [R-...] 日期时间 — 文件与记录序号
```

报告头写入生成时间、手动/自动来源、`manual|scheduled|retry` 触发方式、原始日记范围和运行 ID。

文件先写 `.tmp` 再原子替换固定路径。只有文件交付成功后运行才能标为 `completed`；完成状态写库失败时中控恢复旧文件或删除新文件，避免出现“文件成功但数据库失败”的分裂状态。

## 11. 失败与重跑

Agent 调用、解析、结构校验和审查产物均写入 `agent_artifacts`。若回答不是合法顶层 JSON，中控保存原回答、错误和已有工具遥测，再携带这些信息请求同一 Agent 修订。整个回答只是合法 JSON 外包一层 ` ```json ` 围栏，或完整 JSON 后只有兼容接口误加的孤立引号、结束括号或围栏字符时，可确定性恢复；带解释前缀、两个拼接对象或内部结构损坏仍进入修订，不猜测和补写内容。结构或内容门禁、联网门禁和 Reviewer 审查失败采用相同的最多三次有限修订；普通 API 失败不做内容修订。

每个审计产物保留耗时、HTTP 尝试数、token 用量、缓存 token 和联网证据。第三方检索另存为 `research_search`：选题和查询完全一致时，同输入报告重试可以直接复用它；已完全过审的其他前置阶段也可复用。未过审稿件、输入快照改变、模型改变或不兼容的流水线版本改变时均不命中。

同周期报告可以重跑并覆盖固定 Markdown 文件。每次运行在 SQLite 中仍有独立 ID、输入哈希、来源集合和触发方式。Markdown 报告不是稳定引用对象，运行记录仅用于审计和排障。
