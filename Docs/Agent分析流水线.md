# Agent 分析流水线

本文描述一次分析报告从 Markdown 输入到最终报告的完整生产链。中控实现位于 `analysis.orchestrator`，六类 Agent 位于 `agents/`；Agent 不能互相直接调用，也不能写文件或数据库。

## 1. 总体流程

```text
周期与来源快照
  → Extractor 分块提取 Evidence
  → Cluster 聚合 Theme 和时间关系
  → Explorer 提出 Hypothesis / Insight / 研究请求
  → World 按需生成 Research
  → Reviewer 审查候选节点
  → 中控接受节点与关系
  → Report 生成 Markdown 草稿
  → Reviewer 审查草稿
  → 确定性校验来源
  → 原子保存报告并完成运行
```

所有模型调用统一使用 `current_model`。差异来自中控提供的最小输入、AgentSpec 权限、职责提示和输出校验，而不是 Agent 自己选择模型或权限。

## 2. 运行初始化

中控先把报告类型和锚点日期转换为自然日、周一至周日或自然月范围。输入由以下部分组成：

- 周期内可识别的标准日记记录；
- 本周期显式引用的日记或报告内容；
- 周期开始前的近期日记总结；
- 月报所需的相交周报；
- 周报和月报周期内的每日信息简报（仅作为待重新查证的外部线索）；
- 上一成功同周期运行和近期其他运行的已接受节点。

每条原始记录生成稳定格式的来源 ID `R-YYYYMMDD-NNN`，同时保存相对路径、日期、时间、记录序号、说话者、标签、哈希和最多 500 字符摘录。完整输入快照计算 SHA-256，写入 `analysis_runs`。

## 3. Extractor

月度等较大输入先按字符上限分块。Extractor 是唯一能读取原始记录快照的分析 Agent，输出 `evidence` 节点，来源必须直接引用当前分块提供的 `R-*`。除事件、决定和问题外，它还要保留用户明确表达的观点、思维模型、判断原则、方法论和点子，避免后续流程只剩事件摘要。

中控逐块调用并立即校验、保存。Extractor 不能创建主题、洞见或关系，也不能联网。全部分块结束后如果没有 Evidence，本次运行失败。

## 4. Cluster

Cluster 读取本次 Evidence 和有限的历史 Theme，输出新的 `theme` 及 `member_of`、`evolves_from`、`splits_from`、`merges_into` 关系。

下游 Agent 在 `source_refs`、关系端点和元数据中只看到本次调用的 `K001` 等短别名。中控先映射回持久节点 ID，再根据这些节点已有的 `source_refs`、关系端点和元数据确定性地展开回原始 `R-*`，去重后再校验。不能映射或展开的引用属于未知来源，拒绝保存为候选节点。

Theme 元数据必须声明轨迹：`new`、`continuing`、`growing`、`weakening`、`split` 或 `merged`。

## 5. Explorer 与 World

Explorer 读取本次 Evidence/Theme、可见历史节点和周期内每日信息简报，创建 `hypothesis` 或 `insight`。每个候选必须包含支持节点、可能反证、推断等级、价值说明和是否需要研究。它优先识别和探索观点、思维模型、方法论与未充分展开的点子，并寻找联系、边界、相邻概念和可能方向，而不只是概括记录主题。信息简报不是已接受证据；如果它引出有价值的方向，Explorer 必须创建研究查询，由 World 重新联网查证。

研究查询最多保留五条，中控会去除邮箱、长数字和本地路径等基础私人信息，并把临时目标 ID 映射为持久节点 ID。所有 `research_needed=true` 的候选必须有配套查询。存在可被外部知识实质拓展的内容时通常选择一至三个最高价值方向；只有私人事实或流水记录时可以没有研究请求并跳过 World。

World 读取去隐私查询及对应候选的最小化上下文，并使用被授权的 `web_search`。研究目标使用 `T001` 等短别名，World 必须在 metadata 和关系端点原样复制，中控再映射回持久 ID。它不能把上下文中的私人细节加入搜索词。World 的任务同时包括核查与延伸：查找事实、时效性、支持证据和反例，也查找相关理论、案例、相邻领域、适用边界和不同视角。它输出 `research` 节点和对目标候选的支持或挑战关系；正文应综合说明与原想法的联系、限制和少量探索方向，而不是堆砌链接。声称已核查的结果必须至少带一个 HTTP(S) 来源 URL；否则只能标为 `unverified`。

## 6. 候选审查

Reviewer 接收所有 `candidate` 节点和候选关系，对每个节点且仅对每个节点返回一次决定：`accepted`、`rejected` 或继续保持 `candidate`。中控在模型上下文中用 `N001` 等短别名代替当前候选，用 `H001` 等别名表示关联历史节点，审查完成后再映射回 SQLite ID。

缺项、重复、未知节点或无原因决定会保存为失败产物。中控把校验错误反馈给 Reviewer，并允许完整重做一次审查；第二次仍不合法才使本次运行失败。

接受一个带 `supersedes_id` 的新版本时，旧的已接受版本转为 `superseded`；新版本被拒绝时旧版本保持有效。关系只有在两个端点都被接受或属于可用历史节点时才转为 `accepted`，其余关系拒绝。

## 7. 报告生产与二次审查

Report 只接收通过审查的节点和关系，不读取原始日记，也不能写文件。它返回不含一级标题的 Markdown 正文，并用 `[R-YYYYMMDD-NNN]` 标注依据。报告优先提炼记录中的观点、思维方式、方法论和点子；存在已接受的 Research 时，必须将有价值的核查、反例、相邻概念和延伸方向整合进对应主题，并明确区分用户记录、AI 推断和外部资料。

中控执行两层质量门禁：

1. Reviewer 检查身份混淆、无依据判断、因果越界和表达问题。
2. 确定性代码检查正文非空、没有一级标题或代码围栏、至少引用一个来源且没有未知来源。

首稿不通过时，Report 最多根据审查意见修订一次。仍不通过则运行失败，已有报告不变。

## 8. 原子交付和失败处理

最终文件由中控补充标题、生成方式、周期、运行 ID 和来源索引，先写 `.tmp` 再替换目标报告。如果数据库完成状态写入失败，中控恢复原有报告或删除刚创建的新文件，避免“文件成功、运行状态失败”的分裂结果。

失败时 `analysis_runs` 标为 `failed`。模型调用失败保存失败产物；Agent 响应无法解析为顶层 JSON 对象时，中控先保存失败产物，再允许一次仅修复格式、不重新分析的调用；第二次仍无效则失败。这个机制不重试普通网络或 API 失败。JSON 已解析但权限或结构校验失败时，也保存原始结构化载荷和错误。日志只记录安全的阶段信息，不复制载荷正文。

## 9. 重跑和历史复用

同类型、同周期、同来源方式最近一次成功运行成为 `parent_run_id`。中控可以读取上一运行和近期历史中的已接受节点。只有来源哈希仍一致的上一运行节点才可复用于新报告，避免日记变化后沿用失效判断。

手动和自动报告的 `origin` 不同，因此拥有独立父运行链和固定输出文件。SQLite 保存候选、拒绝、接受、替代和失败历史，使 Agent 工作可以迭代而不是一次性丢弃。

用户通过 `/f` 产生的认可或修正节点作为新的 accepted 历史版本参与后续运行；否决节点不再进入历史上下文。已交付的 Markdown 仍是当时运行的快照，不随反馈回写。
