# Agent 分析流水线

本文描述周报、月报从 Markdown 输入到双板块报告的完整生产链。中控位于 `analysis.orchestrator`，四个职责 Agent 位于 `agents/`。Agent 不能互相直接调用，也不能写文件或数据库。

## 1. 总体流程

```text
周期记录 + 日记引用 + 历史画像 + 每日信息简报
          │
          ├─ Retrospective → 整理与回顾 + 画像候选
          │                         ↓
          │                    Reviewer 审查
          │
          └─ ResearchPlanner → 1~3 个公开研究主题
                                    ↓
                             Researcher 联网研究
                                    ↓
                               Reviewer 审查
                                    ↓
                    中控组装、来源索引、原子交付
```

第一板块由 Retrospective 生成，第二板块由 Researcher 生成。ResearchPlanner 只选择领域与去隐私查询，Reviewer 只做质量门禁。所有 Agent 使用同一 `current_model`；职责差异来自输入、提示、工具权限和输出契约。

## 2. 运行与输入快照

中控只接受 `weekly`、`monthly`。它先计算完整自然周或自然月，并解析周期内标准日记记录。每条记录得到稳定来源 ID：

```text
R-YYYYMMDD-NNN
```

来源目录保存相对文件、日期、时间、记录序号、说话者、标签、内容哈希和最多 500 字符摘录。

运行快照包含：

- 本周期原始记录；
- 本周期记录中明确引用的日记，最多十份且每份最多 12000 字符；
- 周期开始前 30 天的日记总结；
- 月报涉及的同期周报，作为 AI 已有整理而非用户原始观点；
- 周期内每日信息简报，作为待重新核查线索；
- 截止周期结束日仍有效的人物画像。

报告引用目标不会被载入。月报内部读取同期周报不等同于用户引用：它只提供二级整理上下文，最终事实判断仍必须回到 `R-*` 来源。

快照计算 SHA-256 并写入 `analysis_runs`。中控先创建 `running` 运行，再进行任何 Agent 工作。

## 3. 历史画像隔离

历史画像只读取 `status=accepted` 且 `last_observed <= period_end` 的条目，防止以后发生的变化泄漏到过去报告。

SQLite 的 32 位 ID 不直接交给模型。中控在一次调用内将画像映射为 `P001`、`P002` 等短别名，模型输出后再映射回持久 ID。Retrospective 只能替代本次可见画像。

## 4. Retrospective：整理与回顾

Retrospective 可读取原始记录、日记引用、近期总结、月报辅助周报和历史画像，但不能联网。它同时返回：

1. 第一板块 Markdown；
2. 最多 12 条画像候选。

正文必须回顾事实、行动、决定与问题，并整理观点、理念、理想、关注领域和行为模式的变化。每个内容段落必须就近包含合法 `[R-YYYYMMDD-NNN]`，不得把时间先后写成因果，不得心理诊断或给出行为教练式命令。

画像候选只允许：

- `viewpoint`
- `principle`
- `ideal`
- `behavior_pattern`
- `interest`

一次性事件、任务、新闻事实和 AI 建议不能进入画像。每个候选至少有一条本周期记录来源；历史来源可以补充连续性，但不能单独支撑“本期发生更新”。

确定性校验失败或 Reviewer 不通过时，Retrospective 带反馈完整重做一次。第二次仍失败则整次报告失败。

## 5. Reviewer：第一板块门禁

Reviewer 不读取原始记录之外的新材料，也不联网。它检查：

- 事实与时期是否混淆；
- 是否把引用或 AI 内容当成用户立场；
- 来源是否覆盖核心判断；
- 是否出现因果越界、心理诊断、套话或行为教练；
- 每条画像候选是否值得跨周期保留。

Reviewer 必须对每个候选且仅对每个候选返回 `accepted` 或 `rejected`。缺项、重复或未知 ID 会触发一次审查格式重做。

第一板块通过后，中控根据来源目录计算 `first_observed`、`last_observed`。被接受的新版本才会把旧画像改为 `superseded`；被拒绝的候选保留审计记录，但不进入以后上下文。

## 6. ResearchPlanner：研究选题

ResearchPlanner 读取本周期记录、已经通过的第一板块和每日信息简报，选择一至三个公开领域问题。主题来源标为：

- `records`：由记录引出，必须带 `R-*`；
- `news`：由综合信息雷达引出，可以没有日记来源；
- `mixed`：两者共同引出，必须带 `R-*`。

它不能联网，也不生成报告正文。查询在确定性校验前再次清洗邮箱、七位以上长数字、Windows/Unix 本地路径等私人信息。研究题目必须能通过公开资料实质研究，不能把私人细节送入搜索。

## 7. Researcher：领域探索与研究

Researcher 只能看到经过清洗的主题、固定查询和信息线索，不能读取完整原始日记；它是唯一获得 `web_search` 权限的报告 Agent。

第二板块需要：

- 真正执行联网搜索；
- 优先一手、权威、可核查资料；
- 同时寻找支持、反例、边界、相邻概念和不同视角；
- 记录驱动主题保留相应 `R-*`；
- 外部事实就近提供 HTTP(S) Markdown 链接；
- 将探索性推断明确标为推断或假设；
- 不输出新闻链接堆砌或行动指令。

中控同时检查工具遥测、外部来源数组、正文 URL 和记录来源。只声称“已联网”但没有搜索遥测或可验证 URL，不能通过。

## 8. Reviewer：第二板块门禁

Reviewer 单独检查领域研究：

- 来源是否支持正文中的事实；
- 是否只挑支持材料而没有反例或适用边界；
- 记录与研究问题之间是否被夸大；
- 探索性推断是否清楚标注不确定性；
- 是否越权替用户做最终判断。

Researcher 最多根据反馈重新搜索并完整修订一次。第二次仍失败则不交付报告，已有同路径报告保持不变。

## 9. 组装与交付

中控确定性组装：

```markdown
## 一、整理与回顾

...

## 二、领域探索与研究

...

## 来源索引

- [R-...] 日期时间 — 文件与记录序号
```

报告头写入生成时间、手动/自动来源、`manual|scheduled|retry` 触发方式、原始日记范围和运行 ID。

文件先写 `.tmp` 再原子替换固定路径。只有文件交付成功后运行才能标为 `completed`；完成状态写库失败时中控恢复旧文件或删除新文件，避免出现“文件成功但数据库失败”的分裂状态。

## 10. 失败与重跑

Agent 调用、解析、结构校验和审查产物均写入 `agent_artifacts`。若回答不是合法顶层 JSON，中控保存失败产物并允许一次只修复 JSON 格式的调用。结构或内容门禁失败走各板块的一次完整修订；普通 API 失败交给上层自动任务状态处理。

同周期报告可以重跑并覆盖固定 Markdown 文件。每次运行在 SQLite 中仍有独立 ID、输入哈希、来源集合和触发方式。Markdown 报告不是稳定引用对象，运行记录仅用于审计和排障。
