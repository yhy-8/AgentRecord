# 代码与运行机制

本文描述 AgentRecord 当前实现。产品边界见[设计基线](./设计基线.md)，AI 职责和数据流见[各 Agent 功能与联动](./各Agent功能与联动.md)。

## 1. 进程入口

`python main.py`、`python -m AgentRecord` 和 PyInstaller EXE 最终都调用 `AgentRecord.cli.entry.main`。

入口参数：

| 参数 | 行为 |
|---|---|
| 无 | 显示自动化状态并进入交互终端 |
| `--run-automation` | 在独立短进程中检查并执行到期自动任务 |
| `--retry-automation` | 在独立进程中重试全部当前失败自动任务 |
| `--install-automation` | 安装用户级系统调度项 |
| `--uninstall-automation` | 卸载系统调度项 |

Windows 发布同时包含交互用 `AgentRecord.exe` 和无控制台的 `AgentRecordBackground.exe`。计划任务只启动后者，因此每分钟检查不会先创建再隐藏窗口；进程内隐藏仍作为旧发布的兼容兜底。源码版在 Windows 上优先使用同 Python 目录的 `pythonw.exe`。终端模块会把 Windows 标准输出和错误流重配为 UTF-8；构建工作流也设置 UTF-8 环境和 65001 代码页，避免 GitHub Runner 用 CP1252 输出中文 Rich 面板时失败。

## 2. 配置与路径

`settings.py` 是配置边界。源码运行使用仓库根目录 `config.yaml`；冻结运行使用 EXE 同目录配置。`diary_dir`、`analysis_dir`、`log_dir` 的相对路径统一相对于配置文件目录解析。

模型密钥只进入请求头，不得写入日志、报告或测试输出。`/m` 通过临时文件替换更新 `current_model`，同一配置用于总结、手动报告和自动任务。

## 3. 终端与日记写入

`cli.app` 维护记录/报告两种模式。Unix 使用 raw TTY 和 `select`；Windows 直接消费 `ReadConsoleInputW` 的 Unicode `INPUT_RECORD`，包括会令输入句柄保持触发的按键抬起、窗口和焦点等非字符事件，不再用 `kbhit()` 判断原生队列。输入行起点用虚拟终端光标保存/恢复序列锚定，删除和后台通知重绘不估算中英文显示宽度或自动换行行数。同一批到达的可打印字符只刷新一次输出，中文输入法提交长文本时不会逐字符强制刷新。

`journal.py` 是 Markdown 的唯一写边界：

1. 回车提交时捕获一次 `submitted_at`；
2. 用同一时间确定日记文件、文件头和 `HH:MM`；
3. 普通记录以追加方式立即落盘，每条新记录前写入隐藏边界标记，因此多行内容中的 `**HH:MM:**` 不会被误当成另一条记录；
4. 总结只替换 `<summary>`，先写同目录临时文件再原子替换；
5. 引用选择只扫描 `Records/*.md`，写入相对路径和可选关联记录。

追加记录、替换总结和删除最后一条记录共用 `.journal.lock`，避免自动总结原子替换恰好覆盖同时提交的新记录。临时文件使用唯一名称，多进程不共用固定 `.tmp`。

分析上下文解析 `[引用]` 时也只允许日记目录。旧报告引用文本继续保留，但不会加载其目标内容。

## 4. 手动报告与自动重试的并发模型

手动 `/a weekly`、`/a monthly` 交给当前交互进程中的 `ManualReportJobs`：

- 每个窗口最多一个非守护工作线程；
- 工作线程调用公共报告中控，不直接操作终端；
- 完成事件写入线程安全通知队列，由输入主线程显示；
- 当前窗口可以继续记录，但窗口退出会终止这项工作的生命周期。

`/retry` 使用不同机制。交互进程通过 `subprocess.Popen` 启动 `--retry-automation`：Unix 使用新 session，Windows 使用 detached/new-process-group/no-window 标志，标准输入输出均指向空设备。因此当前终端关闭后，重试进程仍独立运行。

子进程在启动时读取失败快照，串行依次重试日总结、人物画像、信息简报、自动周报和自动月报。任何前项仍失败时，当轮停止后续项；未到前项重试门槛时也形成依赖屏障。周/月报告保持 `origin=auto`，运行触发记录为 `trigger=retry`。

## 5. 系统自动任务

类 Unix 安装两条 cron：`@reboot` 和每分钟一次；Windows 安装 `ONLOGON` 与 `MINUTE /MO 1` 两个任务。每个触发器只启动一个短进程，进程完成后退出。安装是幂等刷新：直接替换同名任务，不先卸载。状态检查会比对完整绝对入口和参数，而不是只看任务名或标记。计划项保存安装当时的绝对入口；原路径内代码更新后，新短进程会加载新代码，正在运行的旧进程则完成后退出。移动路径、更换 Python 环境或切换源码版/EXE 版后必须从新入口重新安装。

`run_due_automatic_tasks` 的顺序是：

1. 检查总开关并获取跨进程自动化锁；
2. 写入检查开始时间并读取小时水位与失败重试门槛；
3. 在每个新小时的首次短进程中核对昨日日记总结、昨日人物画像、今日信息简报、上周自动周报和上月自动月报；
4. 今日信息简报另在配置时间到达后立即检测，不必等待下一小时；
5. 对到期的失败任务按失败类别放行，对其余任务不调用模型；
6. 严格按日总结、人物画像、信息简报、周报、月报执行；前项失败就停止，不能继续生成后项；
7. 每个需要生成的任务都重新读取本地输入；报告可从已过审阶段缓存继续；
8. 工作完成后才写入本小时已检测水位，保存完成时间并释放锁。

`.automation-state.json` 通过临时文件替换，只保存调度小时水位、当前执行阶段、失败原因、失败任务的确切日期/周期、重试类别、时间、同输入内容失败次数和不可逆的输入摘要。跨日或跨周期后的重试仍处理原失败目标。它不保存日总结、信息简报、周报或月报的完成进度，也不保存日记正文或 API 密钥。日总结、信息简报和报告由实际内容证明完成；每日人物画像允许零候选，因此由 SQLite 中对应日期的 completed `daily_profile` 运行证明。首次运行新调度器时会删除旧版完成字段与锁屏延后字段。`/status` 是只读展示，不会安装或直接执行任务。

内容或格式首次失败写入下一个整点的 `retry_after`；同一输入第二次仍失败则标记为 `content_blocked` 并停止自动请求。输入、模型或第三方搜索配置变化后，下次整点检测会清除该状态并重新尝试。网络与限流失败写入当前时间后 5 分钟；401/403 鉴权配置错误标记为 `blocked`。`/retry` 忽略时间门槛，可随时执行明确的人工重试。

## 6. 锁屏、休眠、重启与锁

程序不检测桌面锁定状态。锁屏时如果操作系统仍允许运行和联网，自动任务照常执行；如果锁屏使网络断开，请求会被归类为网络失败并在 5 分钟后重试。这样不依赖 Linux `loginctl` 或 Windows 输入桌面等易受环境影响的判断。

休眠或关机期间调度器不运行，也没有错误需要伪造。恢复后，每分钟任务发现当前小时水位尚未写入，会在下一分钟补查。补查仍只覆盖昨日/今日、最近闭合周和最近闭合月，不遍历长期关机期间的历史周月。

具体实现不是监听“唤醒”事件，而是比较 `.automation-state.json` 中的 `last_detection_hour` 与当前 `YYYY-MM-DDTHH`。例如最后一次完成检测是 `08` 时，电脑休眠到 `10:23`，`10:24` 的分钟任务看到当前小时 `10` 与水位 `08` 不同，就执行一次完整缺漏检测并在结束后写入 `10`。水位故意在任务结束后写入；如果进程中途崩溃，下一分钟仍会看到旧水位并从头检查，已经存在的真实产物会自动跳过。

`.automation.lock` 使用文件作为内核锁载体：Unix 为 `flock`，Windows 为 `msvcrt.locking`。进程死亡或系统重启时句柄关闭，内核自动释放锁。文件可以长期存在；文件存在本身不代表任务仍运行，也没有 PID 和超时清理逻辑。

## 7. 模型与网络故障

`ai_client.py` 统一处理 OpenAI 兼容请求和搜索工具授权。连接、DNS、超时、HTTP 408 或服务端 5xx 最多尝试三次，间隔 1 秒、2 秒。HTTP 429 不做秒级连续请求，立即交给自动任务的 5 分钟限流等待。429、401/403、网络错误和内容/格式错误使用不同状态标记，终端不再把它们统一显示成网络错误。

结构化 Agent 在模型配置声明 `json_mode: true` 时发送 `response_format: {"type":"json_object"}`；默认 DeepSeek 配置启用该能力，并设置 32768 的输出上限。日记总结等纯文本任务不使用 JSON mode。其他 OpenAI 兼容接口只有确认支持该参数后才能开启。

每个 Agent 阶段最多进行三次模型尝试。第一次使用完整原请求；JSON、确定性契约或 Reviewer 实质审查失败时，中控把未通过稿件和具体原因作为修订后缀交回同一 Agent，要求保留正确内容并输出完整修订稿。修订后重新执行全部确定性校验和 Reviewer 门禁，不降低质量标准。Reviewer 自身的输出契约错误也使用同样的有限修订，但 Reviewer 的实质否决必须交回原生成 Agent，不能让 Reviewer 代写。

每日信息简报和周/月报是严格审计入口，必须启用第三方搜索。中控逐字执行每个授权查询；缺少已完成查询遥测、来源证据或正文链接与证据不一致都会失败。报告研究的每个查询最多保留 10 条结果，并分配 `W-Q001-001` 形式的证据 ID。Researcher 只看到标题、摘要、日期和证据 ID，不看到 URL，也没有工具权限；它输出 `W-*` 引用后，中控校验主题覆盖、记录来源和证据存在性，再把 ID 确定性渲染为搜索服务返回的精确 URL。Reviewer 只接收成稿实际引用的证据。格式修订和 Reviewer 退回都复用同一证据集，不再搜索。

模型原生搜索仍可用于非严格的通用 AI 调用，但不能代替上述简报和报告入口的第三方搜索。工具循环在真正执行前还会再次对照当前请求的运行时白名单；即使模型伪造未开放的 `tool_call`，也只会收到拒绝结果，不会进入工具实现。

模型 `finish_reason=length` 会被明确归类为输出截断，`content_filter` 归类为输出未完整交付，两者进入同阶段有限修订；`insufficient_system_resource` 按服务端瞬时故障处理。工具参数 JSON 损坏时，错误作为 tool 结果返回模型在当前调用内修正，不直接重启整条流水线。多轮工具调用原样保留服务商返回的 `reasoning_content`；请求不显式发送非必要的 `tool_choice=auto`，避免部分 DeepSeek 思考模式兼容接口拒绝该字段。

修订信息追加在未变化的原始任务与输入之后，尽量保持请求长前缀稳定，便于兼容服务商复用提示缓存。对等输入的失败报告重试还会复用前一运行中已经完成确定性校验和 Reviewer 审核的阶段，以及查询与选题完全一致的 `research_search` 检索证据。缓存键包含输入快照、周期、来源、流水线版本以及非密钥的有效模型/搜索配置（包括 model ID、URL、温度、输出上限和搜索参数），未过审稿件不复用。

结构化 Agent 允许安全移除“整个回答只有一层 JSON 代码围栏”，也允许丢弃完整合法 JSON 后由兼容接口误加的孤立引号、结束括号或围栏字符；不会从解释性文字中猜测 JSON，也不会修补内容结构已经损坏的 JSON。请求次数、耗时、token/缓存 token、搜索查询、重复/拒绝查询、结果与 URL 证据保存到 Agent 审计产物。

失败不会覆盖已有成功报告。结构化 Agent 失败产物与错误写入 SQLite，自动任务类别错误写入状态文件，安全阶段信息写入轮转日志。

## 8. 报告与 SQLite 并发

- 普通记录不访问 SQLite，也不获取自动化锁。
- 模型和网络调用期间不持有 SQLite 事务。
- SQLite 使用 WAL、外键、10 秒 busy timeout 和短 `BEGIN IMMEDIATE` 写事务。
- 手动与自动报告路径分离，分别为 `_manual.md` 和 `_auto.md`。
- 手动报告、自动报告和每日画像共用 `.report.lock`，不会由两个进程同时改写画像状态或报告文件。
- 最终报告先写 `.tmp` 再替换；运行完成写库失败时恢复旧报告或移除新报告，避免文件与运行状态分裂。
- 自动调度和 `/retry` 共用同一个内核锁，不能重叠执行。

## 9. 数据库结构策略

数据库不记录 schema 版本号，也不维护迁移或旧结构兼容代码。空数据库按当前结构创建；已有数据库启动时校验表集合和字段集合，结构不一致就直接报错。程序不自动备份、迁移、覆盖或删除任何数据库文件。确认无需保留后，由用户手动删除主库、`-wal` 和 `-shm`，再由程序重建。

人物画像候选在 Retrospective 与 Reviewer 通过后写入数据库，并关联状态仍为 `running` 的运行。每日画像运行只处理前一日原始记录，不生成 Markdown 日报；周期报告仍可根据更长周期证据修订画像。只有所属运行是 `completed` 的候选才能进入画像时间截面，所以每日审查、后续研究或文件交付失败时，候选虽然作为审计留存，却不会进入后续上下文，也不会替代旧画像。运行成功时，同一个完成事务把运行设为 `completed`，并把被新版替代的旧画像设为 `superseded`。按历史周期查询时，中控会按所属运行周期和反馈日期重建当时仍有效的版本，以后的替代、修正或否决不会改写过去的输入快照。

`Records/` 不属于分析数据库版本处理范围，数据库初始化失败不得改写或删除日记。

## 10. 验证清单

修改入口、自动化、日记或报告中控后至少验证：

```bash
python -m py_compile $(rg --files AgentRecord tests -g '*.py')
python -m unittest discover -s tests -v
```

Windows 发布还需验证两个 PyInstaller 单文件构建、EXE 同目录配置、计划任务指向 `AgentRecordBackground.exe`，以及两个后台入口都不创建窗口。GitHub Actions 在每次 `push` 和手动 `workflow_dispatch` 时执行测试并构建 Windows 产物。
