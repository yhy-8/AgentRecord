# 代码与运行机制

本文描述 AgentRecord 当前实现。产品边界见[设计基线](./设计基线.md)，报告内部流程见 [Agent 分析流水线](./Agent分析流水线.md)。

## 1. 进程入口

`python main.py`、`python -m AgentRecord` 和 PyInstaller EXE 最终都调用 `AgentRecord.cli.entry.main`。

入口参数：

| 参数 | 行为 |
|---|---|
| 无 | 显示自动化状态并进入交互终端 |
| `--run-automation` | 在独立短进程中检查并执行到期自动任务 |
| `--retry-automation` | 在独立进程中重试全部当前失败自动任务 |
| `--install-automation` | 安装用户级系统调度项 |
| `--uninstall-automation` | 卸载系统调度项 |

Windows 打包程序只在两个自动化入口隐藏控制台，普通交互入口仍显示窗口。

## 2. 配置与路径

`settings.py` 是配置边界。源码运行使用仓库根目录 `config.yaml`；冻结运行使用 EXE 同目录配置。`diary_dir`、`analysis_dir`、`log_dir` 的相对路径统一相对于配置文件目录解析。

模型密钥只进入请求头，不得写入日志、报告或测试输出。`/m` 通过临时文件替换更新 `current_model`，同一配置用于总结、手动报告和自动任务。

## 3. 终端与日记写入

`cli.app` 维护记录/报告两种模式。Unix 使用 raw TTY 和 `select`，Windows 使用 `msvcrt`，从而在后台通知出现后恢复用户正在输入的内容。

`journal.py` 是 Markdown 的唯一写边界：

1. 回车提交时捕获一次 `submitted_at`；
2. 用同一时间确定日记文件、文件头和 `HH:MM`；
3. 普通记录以追加方式立即落盘；
4. 总结只替换 `<summary>`，先写同目录临时文件再原子替换；
5. 引用选择只扫描 `Records/*.md`，写入相对路径和可选关联记录。

分析上下文解析 `[引用]` 时也只允许日记目录。旧报告引用文本继续保留，但不会加载其目标内容。

## 4. 手动报告与自动重试的并发模型

手动 `/a weekly`、`/a monthly` 交给当前交互进程中的 `ManualReportJobs`：

- 每个窗口最多一个非守护工作线程；
- 工作线程调用公共报告中控，不直接操作终端；
- 完成事件写入线程安全通知队列，由输入主线程显示；
- 当前窗口可以继续记录，但窗口退出会终止这项工作的生命周期。

`/retry` 使用不同机制。交互进程通过 `subprocess.Popen` 启动 `--retry-automation`：Unix 使用新 session，Windows 使用 detached/new-process-group/no-window 标志，标准输入输出均指向空设备。因此当前终端关闭后，重试进程仍独立运行。

子进程在启动时读取失败快照，依次重试日总结、信息简报、自动周报和自动月报。周/月报告保持 `origin=auto`，运行触发记录为 `trigger=retry`。

## 5. 系统自动任务

类 Unix 安装两条 cron：`@reboot` 和每分钟一次；Windows 安装 `ONLOGON` 与 `MINUTE /MO 1` 两个任务。每个触发器只启动一个短进程，进程完成后退出。

`run_due_automatic_tasks` 的顺序是：

1. 检查总开关和会话锁定状态；
2. 获取跨进程自动化锁；
3. 写入检查开始时间；
4. 依次检查日总结、每日信息、已闭合周报和已闭合月报；
5. 每项成功后才推进该项进度；
6. 保存完成时间并释放锁。

`.automation-state.json` 通过临时文件替换，保存各任务最后成功位置、失败原因、锁屏延后时间和重试时间。`/status` 是只读展示，不会安装或直接执行任务。

## 6. 锁屏、休眠、重启与锁

`analysis.session_state` 提供尽力而为的桌面锁定检测：

- Linux：通过 `loginctl` 读取当前 UID 会话的 `Active` 和 `LockedHint`；
- Windows：尝试打开并切换输入桌面；
- 无法判断时返回未锁定，避免检测能力缺失导致系统永久停摆；
- 测试或特殊部署可用 `AGENTRECORD_SESSION_LOCKED=0|1` 覆盖。

自动化在进程开始、任务之间和每次自动模型/搜索请求前检查锁定。锁定时不再发出请求，并写入“解锁后下一分钟内重新检查”。休眠时调度器不会运行；恢复后等待下一分钟触发。重启时 `@reboot`/`ONLOGON` 与每分钟任务共同补查。

`.automation.lock` 使用文件作为内核锁载体：Unix 为 `flock`，Windows 为 `msvcrt.locking`。进程死亡或系统重启时句柄关闭，内核自动释放锁。文件可以长期存在；文件存在本身不代表任务仍运行，也没有 PID 和超时清理逻辑。

## 7. 模型与网络故障

`ai_client.py` 统一处理 OpenAI 兼容请求和搜索工具授权。连接、DNS 或超时异常最多尝试三次，间隔 1 秒、2 秒；收到 HTTP 响应后的业务错误不在这一层盲目重试。

自动请求运行在 `automatic_request_mode` 上下文中。若 Agent 执行过程中发生锁屏，下一次模型或搜索调用会被拒绝，任务保持未成功状态。

失败不会推进任务进度，也不会覆盖已有成功报告。结构化 Agent 失败产物与错误写入 SQLite，自动任务类别错误写入状态文件，安全阶段信息写入轮转日志。

## 8. 报告与 SQLite 并发

- 普通记录不访问 SQLite，也不获取自动化锁。
- 模型和网络调用期间不持有 SQLite 事务。
- SQLite 使用 WAL、外键、10 秒 busy timeout 和短 `BEGIN IMMEDIATE` 写事务。
- 手动与自动报告路径分离，分别为 `_manual.md` 和 `_auto.md`。
- 最终报告先写 `.tmp` 再替换；运行完成写库失败时恢复旧报告或移除新报告，避免文件与运行状态分裂。
- 自动调度和 `/retry` 共用同一个内核锁，不能重叠执行。

## 9. 数据库重定向

当前 schema 为 v3。遇到非 0、非 3 的旧数据库时，程序使用 SQLite backup API 生成 `.analysis.sqlite3.v<版本>.legacy.bak`，随后移除旧主库、WAL、SHM 并创建新库。

这是有意的产品重定向：不迁移旧通用知识图谱，但不触碰 `Records/`。v3 保存报告运行、Agent 产物、来源目录、人物画像和用户反馈。

## 10. 验证清单

修改入口、自动化、日记或报告中控后至少验证：

```bash
python -m py_compile $(rg --files AgentRecord tests -g '*.py')
python -m unittest discover -s tests -v
```

Windows 发布还需验证 PyInstaller 单文件构建、EXE 同目录配置、计划任务创建与两个后台入口隐藏窗口。
