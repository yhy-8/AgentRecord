# 代码与运行机制

本文描述 AgentRecord 当前代码如何启动、接收输入、写入数据、执行后台工作并交付结果。产品取舍和不可破坏的边界见[设计基线](./设计基线.md)。

## 1. 进程入口

源码入口 `main.py`、`python -m AgentRecord` 和 PyInstaller EXE 最终都调用 `AgentRecord.cli.entry.main`。

入口先加载 `config.yaml` 并配置轮转日志，然后按参数分流：

- 无进程参数：检查并显示系统自动任务安装状态，然后进入交互主循环。
- `--run-automation`：执行一次到期任务检查后退出。
- `--install-automation`：安装操作系统调度项。
- `--uninstall-automation`：卸载调度项。

Windows 打包程序仅在 `--run-automation` 模式隐藏控制台，普通交互入口不隐藏窗口。

## 2. 配置与路径

`settings.py` 是配置边界。源码运行时以仓库根目录的 `config.yaml` 为准；冻结运行时以 EXE 同目录的配置为准。`diary_dir`、`analysis_dir` 和 `log_dir` 可以使用绝对路径，相对路径统一相对于配置文件目录解析。

模型密钥只从配置进入请求头，不得写入日志、报告、测试输出或提交历史。`ModelConfig.select` 只替换 `current_model` 行，通过临时文件替换保留原有注释和排版。

## 3. 交互循环与模式隔离

`cli.app.run_interactive` 保存当前模型和当前模式。输入提交后只捕获一次本地时间，并把同一个时间传给日记层，避免跨午夜时文件名、文件头和记录时间不一致。

记录模式只接受记录、查看、引用、删除和清屏；报告模式只接受总结、报告、节点反馈和模型切换。`/status` 是两种模式共用的只读命令。普通文字在报告模式不会落盘，报告命令在记录模式也不会执行。

终端输入没有使用内置 `input()`：Unix 通过 raw TTY 和 `select` 读取 UTF-8 字符，Windows 通过 `msvcrt` 读取按键。这层负责退格、宽字符显示、控制键和异步通知后的输入行重绘。

## 4. 原始日记写入

`journal.py` 是 Markdown 原始记录的唯一写边界。

普通记录的流程是：

1. 用户按回车时捕获 `submitted_at`。
2. 由该时间确定 `Records/YYYY-MM-DD.md`。
3. 文件不存在时写入固定文件头、`<summary>` 和“原始记录流”标题。
4. 以追加模式一次写入带 `HH:MM` 的记录。

总结是唯一允许程序改写的日记区域。更新时先读取完整文件，只替换 `<summary>` 内容，写入同目录临时文件后原子替换。模型或写入失败不得改变原文件。

引用仍是用户原始记录：程序写入来源类型、周期、相对链接和可选关联想法，但不修改被引用文件。

## 5. 手动报告后台线程

手动 `/a` 在确认目标周期和覆盖行为后，将任务交给当前窗口的 `ManualReportJobs`。周报或月报未指定日期时，命令层扫描日记文件并计算最近自然周期选项。每个窗口只有一个工作线程：

1. 主线程复制当前模型配置并启动非守护线程。
2. 界面立即回到记录模式，继续处理普通记录。
3. 工作线程调用公开的 `generate_analysis_report`，自身不实现分析逻辑。
4. 完成或失败后只向终端通知队列提交一条事件。
5. 输入主线程定期取出事件，清除当前输入行、显示通知，再恢复提示符和已输入字符。

工作线程不得直接调用 Rich 或写标准输出，否则会破坏 raw TTY 状态和用户正在输入的内容。

程序能拦截 Ctrl+C、Ctrl+D 等软退出：有手动报告运行时取消退出并提示等待。操作系统关闭窗口或终止进程无法通过同一套跨平台逻辑可靠阻止，因此启动时明确提示不要关闭。当前不持久化线程任务，也不在新进程恢复未完成的手动报告。

## 6. 系统自动任务

系统自动任务是独立短进程，不是交互线程。Windows 任务计划程序和类 Unix cron 每小时触发一次 `--run-automation`；进程检查到期工作、保存状态后退出。

交互启动时会只读检查两个 Windows 计划任务或两条 cron 记录是否完整存在，并显示已安装、未安装或安装不完整。状态文件还保存检查开始/完成时间，`/status` 同时展示三类进度和下次小时检查将重试的失败。该检查不安装任务，也不在交互进程中执行自动分析。

`AnalysisReports/.automation.lock` 使用排他创建，防止登录触发、重启触发和整点检查彼此重叠。锁只约束自动调度进程，交互程序和原始记录写入不检查它。锁超过六小时才视为可能遗留。

任务按日总结、周报、月报检查。每项成功位置和失败信息保存在 `.automation-state.json`，状态通过临时文件替换。自动周报和月报只处理闭合周期，手动和自动报告使用不同文件名。

## 7. 并发与锁边界

- Markdown 普通记录不访问 SQLite，也不获取自动任务锁。
- 自动日总结处理昨天，前台普通记录写今天，正常路径不会同时改同一日记。
- 模型调用期间不持有 SQLite 事务；每个产物、节点或关系只在短事务内落库。
- SQLite 使用 WAL、外键、10 秒忙等待和 `BEGIN IMMEDIATE` 串行化写事务。
- 手动报告线程与自动任务进程可能同时分析，但 `_manual.md` 与 `_auto.md` 路径隔离；SQLite 负责短写事务竞争。
- 同一窗口禁止同时启动两份手动报告。不同交互窗口仍可能各自启动任务，当前没有全局手动任务锁。

## 8. 日志与故障边界

应用日志写入 `log_dir/AgentRecord.log`，单文件 5 MiB，保留两个归档。日志记录进程动作、运行 ID、Agent 阶段和结构错误，不记录日记全文、模型请求正文、模型响应正文或 API 密钥。

模型、网络、分析、数据库或报告审查失败不得阻止原始记录追加，也不得覆盖已有报告。结构化 Agent 输出即使校验失败，也作为 `failed` 产物保存到 SQLite 供排障。

## 9. 打包影响

根入口必须保持极薄，避免源码、模块入口和 EXE 出现不同启动逻辑。修改入口、包名或依赖后，应同时验证：

- `python main.py`；
- `python -m AgentRecord`；
- GitHub Actions 的 Windows 测试；
- PyInstaller 单文件构建；
- EXE 同目录配置解析；
- Windows 后台任务隐藏控制台的条件。
