# AgentRecord 设计基线

> 状态：架构基线，确认于 2026-07-15。本文是项目定位、数据边界和代码架构的唯一设计依据；README 只说明使用方法。

## 1. 项目定位

AgentRecord 是一个本地优先的个人记录与探索性分析系统。

用户记录的主体是日常事实、做过和计划做的事、观点、判断、问题与灵感。记录本身应当像普通日记一样低摩擦、可靠、可直接阅读。AI 的价值不在于随时回答常识问题或陪用户聊天，而在于跨越零散记录，发现值得继续思考的关联、变化、矛盾、盲点和研究方向，并将结果整理成独立报告。

核心价值链是：

```text
自然记录 → 可靠保存 → 引用与追溯 → 周期性整理 → 发散探索 → 独立报告
```

系统的目标不是替用户结束思考，而是提高下一轮思考的质量。

### 明确不做

- 不提供 `@AI` 即时聊天入口；知识问答、常识咨询和开放聊天交给现成 AI 产品。
- 不把日记变成结构化表单，不强制用户预先分类。
- 不让 AI 改写用户原文或替用户确认长期观点。
- 不把每条记录自动变成任务，也不为了展示 AI 能力制造弱关联。
- 不在需求形成前建设自由讨论式多 Agent、复杂 GUI 或专用向量数据库。

## 2. 产品原则

### 2.1 捕获优先

普通输入立即追加到当天 Markdown 日记。断网、模型失败或后台分析失败都不能影响原始记录保存。

“当天”和记录时间以用户最后按下回车、提交完整内容的时刻为准，而不是开始输入的时刻。一次提交只能捕获一个本地时间值，并用它同时决定日记文件名、文件头和记录行时间；跨午夜输入不得拆分，也不得因多次读取时钟形成日期不一致。

### 2.2 原文与派生结果分离

用户原话是最高优先级的事实源。总结、标签、关系、索引和报告都是派生结果，必须可追溯、可重建、可否决。AI 不能静默覆盖原始记录。

日记顶部 `<summary>` 是唯一允许程序更新的日记区域；更新必须通过 `journal.py`，采用临时文件替换。分析报告保存在独立目录。

### 2.3 报告优先

AI 的主要交付物是日记总结、周报、月报以及未来的专题报告，不是聊天记录。手动日报用于需要当天下钻的场景。报告应关注少量有依据的强关联、思想演化、潜在矛盾和待判断问题，能够脱离终端会话独立阅读。

### 2.4 人保留判断权

AI 可以提出候选关联和反例，但必须区分“用户记录”“外部事实”和“AI 推断”。最终方向、价值判断和现实决策由用户完成。

### 2.5 简单且可追溯

能用 Markdown 明确表达的关系，不提前引入数据库。只有当跨记录关系、检索规模和反馈闭环证明有需要时，才增加可重建的派生索引。

## 3. 交互基线

```text
普通文字                   立即追加到今日日记
/v [日期]                  查看日记
/ref [类型] [筛选词]       选择并引用日记、日报、周报或月报
/s [日期]                  生成指定日记顶部总结
/a daily [日期]            生成分析日报
/a weekly [日期]           生成日期所在自然周的分析周报
/a monthly [日期]          生成日期所在自然月的分析月报
/m                         切换总结和分析使用的模型
/d                         删除今日最后一条记录
/c                         清空终端显示
/h                         显示帮助
```

`/ref` 支持 `diary`、`daily`、`weekly`、`monthly`，也接受对应中文名称。未给类型时由终端询问；系统展示最近 20 个匹配来源，选择后可以在同一条引用记录中补充由该来源引出的新想法。

输入以 `@` 开头时不再触发模型，它和其他普通文字一样被原样记录。这避免旧习惯导致内容丢失，同时彻底取消即时聊天语义。

## 4. 引用格式

引用是原始记录流中的一种用户记录，不是 AI 派生数据。标准格式为：

```markdown
**14:32 [引用]:** [分析周报 | 2026-07-06 至 2026-07-12](<../AnalysisReports/Weekly/2026-07-06_to_2026-07-12.md>)

这里记录由该报告继续展开的新想法；可以留空。
```

格式语义：

- 当前日记文件名给出引用日期，行首 `HH:MM` 给出引用时间。
- `[引用]` 是稳定类型标签，便于人和后续分析程序识别。
- 链接文本同时写明来源类型和周期，即使链接失效仍可理解。
- 链接使用相对路径和 `/` 分隔符，兼容 Windows 与类 Unix 环境并便于整体迁移。
- 被引用的文件不会被修改；删除引用只删除当前日记中的这条记录。
- 可选关联记录与引用位于同一条记录内，避免失去上下文。
- 生成分析报告时，程序会读取本周期最多 10 个去重引用来源，每个最多提供 12000 字符；只允许读取日记和报告目录内的 Markdown，不递归追踪引用。

旧日记无需迁移。引用能力只增加新的标准记录，不改变既有文件头、`<summary>`、分隔线和普通记录格式。

## 5. 数据分层

```text
原始层
  Records/YYYY-MM-DD.md
  ├─ <summary>：程序生成的当日摘要
  └─ 原始记录流：普通记录与 [引用] 记录

派生交付层
  AnalysisReports/
  ├─ Daily/YYYY-MM-DD.md
  ├─ Weekly/YYYY-MM-DD_to_YYYY-MM-DD.md
  ├─ Monthly/YYYY-MM.md
  └─ Topics/                       # 未来专题报告

可重建索引层                       # 未来、按实际需求加入
  Record → Thought → Hypothesis → Theme → Research → Insight → Report
```

### 原始层

- 每天一个 UTF-8 Markdown 文件。
- 用户内容只追加，除用户明确执行删除外不回退。
- AI 不能直接写原始记录流。
- 总结更新失败不得损坏已有文件。

### 派生交付层

- 日报、周报和月报使用确定文件名，重复生成覆盖同一路径，不无故制造副本。
- 报告记录生成时间和输入日期范围。
- 模型或网络失败不改变原始日记。

### 可重建索引层

当前不实现。未来首先考虑 SQLite 与 FTS5；稳定 ID、标签、关系、向量和反馈均保存在日记之外，并提供从 Markdown 全量重建的规则。

## 6. 代码架构

当前保持五个有实际职责的模块，不再为终端单独保留 `cli.py`：

```text
main.py                 唯一入口；终端输入、命令解析与结果展示
  ├─ journal.py         原始日记唯一读写边界；日期、引用和总结区域
  ├─ analysis.py        总结、报告、自动调度及未来分析编排
  │   ├─ journal.py
  │   └─ ai_client.py
  └─ settings.py        模型选择与目录配置

ai_client.py            OpenAI 兼容协议、只读工具循环与联网搜索
settings.py             被各层共享，不依赖业务模块
```

边界规则：

- `main.py` 只映射交互，不保存分析提示词，不直接写 Markdown。
- `journal.py` 是原始日记的唯一写入边界；分析代码和未来 Agent 只能调用它公开的受控接口。
- `ai_client.py` 只处理模型协议、只读日记工具和搜索，不决定何时生成何种报告。
- `analysis.py` 准备上下文、定义分析任务、调用模型、验证成功并保存派生产物。
- `settings.py` 只负责配置。API 密钥不得进入日志、报告、测试输出或提交历史。

未来分析能力先进入 `analysis.py`。只有某类 Agent 已形成独立输入输出、状态和足够实现时，才拆入 `agents/` 包，避免空壳抽象。

## 7. 分析流程

```text
日常记录与显式引用
        │
        ▼
Markdown 原始记录（立即保存）
        │
        ├── 跨日总结 → 原日记 <summary>
        │
        └── 探索性分析
              ├─ 本周期原始记录
              ├─ 之前 30 天总结
              ├─ 引用关系
              └─ 必要时外部检索
                       │
                       ▼
             日总结 / 周报 / 月报
```

### 7.1 周期闭合与自动触发

周期只按运行机器的本地日历计算：自然日为 00:00–23:59，自然周为周一至周日，自然月为每月 1 日至最后一日。未闭合的周期绝不自动生成，以免分析仍在变化的输入。

自动任务并非操作系统常驻服务：

1. 程序启动时立即检查。
2. 程序运行期间每小时检查；临近午夜时在跨过 00:00 约 5 秒后检查。
3. 日总结最早在次日 00:00 后生成，写回前一日日记的 `<summary>`。
4. 周报最早在下周一 00:00 后生成，覆盖上一完整周一至周日。
5. 月报最早在下月 1 日 00:00 后生成，覆盖上一完整自然月。

默认自动流程是“日总结 + 周报 + 月报”。日报只保留手动下钻，因为日总结已经承担每日压缩，再自动生成日报会增加重复产物和模型消耗。

调度按日总结、周报、月报的顺序执行，但三者保存独立成功位置、失败信息和重试进度。单项失败不会回退已有产物，也不会阻断其他可以直接基于原始记录运行的周期任务。失败任务在下次启动或下一小时检查时重试。

首次启用只处理各自最近一个闭合周期，避免对全部历史记录产生意外成本；形成状态后，从上次成功位置连续补做程序关闭期间遗漏的周期。无记录周期只推进检查位置，不生成空文件。状态保存在 `AnalysisReports/.automation-state.json`。

固定路径保证幂等：相同周期不会因重复检查产生副本，手动重新生成则明确覆盖同一路径。

### 日报关注

- 当天重要的新想法、变化和注意力中心。
- 引用所形成的明确延伸关系。
- 与历史记录少量、可追溯的强关联。
- 潜在矛盾、盲点和少量值得继续判断的问题。

### 周报关注

- 本周主要思想簇及其变化。
- 新出现、持续强化或衰退的主题。
- 反复出现但未解决的问题。
- 观点之间的矛盾、转变和可能原因。
- 下一周期最值得继续探索的少量方向。

### 月报关注

- 一个月内注意力和精力如何分配，哪些领域实际占据了持续关注。
- 长期主题的出现、增强、衰退、分叉或合并。
- 重要判断在后续记录或外部事实中得到的支持、挑战和修正。
- 跨周反复出现的行为、问题、限制条件和潜在盲点。
- 下个月最值得继续探索的少量方向，而不是完整任务清单。

月报综合当月原始记录、显式引用、月前 30 天总结，以及与该月相交的已有周报。下级报告只作为 AI 分析材料，不能被表述成用户已经认可的观点。这里先建立可靠的周期输入与产物边界；更复杂的提取、研究、聚类和审查编排仍属于 `analysis.py` 的后续核心设计。

## 8. 演进路线

当前基线先验证“自然记录—显式引用—周期报告”是否真正有用。之后按证据演进：

1. 改进报告对 `[引用]` 关系的识别和呈现。
2. 当字面检索不足时，增加可重建的 SQLite/FTS 索引。
3. 从原始 Record 提取 Thought，再逐步形成 Hypothesis、Theme、Research 和 Insight。
4. 在用户能确认、否决和修正候选关联后，再引入关系权重与专题报告。
5. 仅在职责足够独立时拆分 Extractor、Explorer、Cluster、Reviewer、World、Report Agent。

每一步都必须保持：旧日记可无损读取、派生数据可重建、AI 失败不影响记录、报告引用能够追溯到来源。
