# AgentRecord：个人思维智库升级设计

> 状态：V2 第一阶段功能基线，记录于 2026-07-14。本文同时记录已确认设计、已经实现的能力、后续计划和待讨论事项。

## 1. 升级目标

AgentRecord 当前是“追加式日记 + 即时 AI 问答”。升级目标是在保留这一核心体验的基础上，增加一个本地优先的个人思维智库层：用户继续自然记录和随时咨询 AI，系统另行自动总结日记、分析思想并生成每日、每周报告。

核心变化不是增加更多聊天命令，而是改变价值交付方式：

`原始日记与即时问答保持不变 + 后台加工 + 独立分析报告`

这里的“不要流式对话”包含两层含义：

1. 技术上不必逐 token 展示生成过程，只展示完成状态和最终产物。
2. 产品上，自动分析任务不要求用户陪 AI 多轮推进；但用户通过 `@` 主动咨询时，仍然立即获得正常问答回复。

### 已确认的需求基线

- `@` 保持现有即时问答功能和日记记录方式，不改造成报告触发器。
- 删除 `/q`，历史查阅和咨询统一由 `/v` 与 `@` 承担。
- `@` 使用的 AI 不再拥有修改日记总结的工具。
- 日记总结是每天 Markdown 文件开头 `<summary>` 中的当日内容摘要，可自动或手动生成。
- 跨过零点后自动总结前一天；程序中断时允许下次启动补做。
- 分析报告是 AI 对思想进行关联、挖掘和审视后的独立产物，不属于原始日记。
- 分析日报、周报均支持自动生成和手动生成，并保存到独立文件夹。
- 每天一个 Markdown 文件及其现有内容格式完全不变，兼容全部旧日记。
- 手动命令调整为删除 `/q`，增加 `/s` 与 `/a`；命令精简留待功能稳定后再讨论。

### 三类 AI 产物的边界

| 类型 | 触发方式 | 保存位置 | 作用 |
|---|---|---|---|
| 即时问答 | `@问题` | 按现有格式写入当日日记 | 回答咨询或即时执行任务 |
| 日记总结 | 跨日自动或 `/s` | 原日记顶部 `<summary>` | 概括单日原始日记 |
| 分析报告 | 定期自动或 `/a` | `AnalysisReports/` | 跨记录关联、挖掘、质疑和高层审视 |

## 当前阶段进度（2026-07-14）

### 已经明确的设计需求

#### 产品定位

- AgentRecord 从简单 AI 日记升级为个人思维智库或个人智能系统。
- 日记是自然输入入口，不是最终产品；核心价值是增强探索、总结、聚类、研究、审查和汇报能力。
- 系统目标不是替用户结束思考，而是提高下一轮思考的质量。
- 人负责提出直觉、选择方向、评价价值和做最终决策；AI 负责提取、展开、关联、研究、质疑和汇报。
- 输入保持低摩擦和人性化，输出应高度结构化。

#### 交互与自动化

- 普通文字继续直接写入当天日记，不要求用户填写类型、领域或标签。
- `@` 保持即时 AI 问答功能，允许咨询、分析和工具调用，不改造成异步报告命令。
- 自动分析以最终报告为主要交付形式，不要求用户参与冗长的流式多轮对话；AI 在确有必要时可以提问。
- 删除价值有限且会额外调用模型的 `/q`。
- 日记总结采用跨日自动生成和 `/s` 手动生成两种方式。
- 分析报告采用每日、每周自动生成和 `/a` 手动生成两种方式。

#### 数据边界

- 继续使用每天一个 Markdown 文件的原始日记形式。
- 现有文件名、`<summary>`、分隔线和原始记录流格式不改变，兼容历史日记。
- 日记顶部总结属于原始日记文件；分析报告属于独立产物，必须保存在不同目录。
- `@` 问答仍按现有格式记录到当天日记，但即时问答 AI 无权修改 `<summary>`。
- 未来标签、思想对象、聚类、关系和向量等信息属于派生数据，不能污染或改写原始日记。

#### 技术与架构

- 只保留 OpenAI 兼容接口，不再维护 Gemini 专用协议。
- 代码按有实际意义的职责拆分，不能继续堆积在单个 `main.py` 中，也不能制造大量功能过少的文件。
- 分析层必须方便以后接入 Extractor、Explorer、Cluster、Reviewer、World、Report 等 Agent。
- 长期分析对象链确定为：`Record → Thought → Hypothesis → Theme → Research → Insight → Report`。

### 打算实现的需求

以下内容属于后续路线，不代表已经完成。

#### 思想数据层

- 在不改变原始日记格式的前提下，建立可重建的派生数据库。
- 从原始 Record 中提取 Thought，包括观点、问题、直觉、情绪和隐含假设。

#### Agent 分析层

- Explorer Agent：拓宽思考空间，提出相关领域、可能性和待验证问题。
- Cluster Agent：跨日期聚类长期主题，发现重复、汇聚和思想演化。
- Reviewer Agent：寻找逻辑漏洞、历史反例、遗漏变量和过度推断。
- World Agent：连接新闻、论文、产品、公司和市场变化，持续验证个人观点。
- Report Agent：综合其他 Agent 结果，生成可独立阅读的最终报告。

#### 报告和反馈层

- 提升日报和周报，使其基于结构化 Thought、Theme 和 Research，而不仅是直接分析原始文本。
- 增加专题研究报告、月度认知报告、项目推进报告、机会发现和风险提醒等报告类型。
- 允许用户接受、修改、否决或继续探索 AI 提出的观点和关联。
- 根据用户反馈调整主题权重、关联可信度和后续研究方向。
- 研究更合适的报告查看、完成提醒和失败重试体验。

### 当前明确不做的事项

- 不改变用户自然记录习惯，不强制填写结构化表单。
- 不让 AI 自动改写原始记录或替用户确认价值判断。
- 不立即创建多个只有提示词或空接口的 Agent 文件。
- 不优先建设复杂 GUI、专用向量数据库或自由讨论式多 Agent 系统。
- 不为了表现 AI 能力而生成弱关联、重复摘要或把所有思想转成任务。

## 2. 改造前项目评估

### 可以保留的基础

- Markdown 本地记录简单、透明、便于备份。
- 原始记录由程序追加，AI 只能更新 `<summary>`，已经具备初步的数据边界意识。
- 模型请求采用 OpenAI 兼容接口，便于在同一协议下切换模型服务。
- 已有日期读取、历史搜索、联网搜索和只读查阅模式。
- 当前 API 调用本身是等待完整响应后展示，并非 token 流式输出。

### 与目标之间的差距

- 交互仍是“输入问题—阻塞等待—显示回答”的同步聊天模式。
- 用户原文、`@AI` 问题和 AI 回答混在同一日志流中，后续检索可能把 AI 生成内容误当成用户思想。
- 历史搜索只有字面子串匹配，无法处理同义表达、主题聚类或思想演化。
- `<summary>` 只覆盖当日并依赖手动触发，没有自动日报、周报和专题档案。
- 记录没有外部索引 ID，难以可靠建立引用、关系、反馈和增量索引；该 ID 应保存在派生索引中，不能改变原始日记格式。
- 没有后台任务队列、定时调度、失败重试、幂等和报告状态。
- 改造前 `main.py` 同时负责配置、存储、模型适配、工具、终端 UI 和流程编排，继续增加功能会加速耦合。
- `/q` 通过第二次模型调用生成一句“轻记录”，成本不低，却缺少来源、置信度和长期知识价值。

## 3. 产品原则

### 3.1 捕获优先

普通输入立即落盘，不要求用户先选择“灵感、任务还是观点”。分类是 AI 的工作。AI 故障、断网或后台任务失败都不能影响记录。现有每天一个 Markdown 文件及其中的内容格式保持不变，以兼容全部历史日记。

### 3.2 原文与解释分离

用户原话是最高优先级的事实源。标签、摘要、关联、聚类、向量和报告都是派生层，必须可追溯、可重建、可否决。

### 3.3 报告优先

AI 的主要交付物是可独立阅读的最终报告，不是长聊天记录。中间搜索、工具调用和推理过程默认折叠，只保留必要的来源和方法说明。

### 3.4 AI 主动，但不过度打扰

AI 可以主动发现关联和问题，但只在报告中呈现高价值结果。弱关联、重复复述和把所有想法都转成待办，会迅速制造噪声。

### 3.5 人保留判断权

AI 可以提出“这两条思想可能矛盾”，但不能把它直接写成用户已经改变了立场。长期观点、决策和价值判断必须由用户确认。

## 4. 人与 AI 的分工

| 阶段 | 用户 | AI |
|---|---|---|
| 捕获 | 表达灵感、判断、疑问、观察和资料 | 自动识别记录类型、主题、实体和时间信息 |
| 整理 | 必要时纠正 | 去重、聚类、命名主题、建立候选关系 |
| 研究 | 决定问题是否值得投入 | 查资料、核实事实、寻找反例和不同视角 |
| 审视 | 做价值判断和最终取舍 | 发现重复、演化、矛盾、盲点和长期模式 |
| 交付 | 阅读、确认、否决、收藏 | 生成每日、每周和专题最终报告 |
| 长期记忆 | 确认哪些结论真正代表自己 | 维护可追溯的派生知识和关系权重 |

适合让 AI 自动决定的是低风险、可撤销、可重建的事项。会改变用户原文、长期立场或现实行动的事项，需要用户确认。

## 5. 建议的整体流程

```text
普通记录 / @ 即时咨询 / 外部资料
                  │
                  ▼
         原始记录层：立即追加保存
                  │
                  ▼
      后台加工层：解析、索引、分类、实体
                  │
          ┌───────┴────────┐
          ▼                ▼
   历史语义召回       必要时联网研究
          └───────┬────────┘
                  ▼
       关系层：聚类、演化、矛盾、缺口
                  │
                  ▼
   交付层：日记总结 / 日报 / 周报 / 专题报告
                  │
                  ▼
       用户确认、否决、收藏或形成新记录
```

后台加工可以分级，以控制成本：

- L0，立即执行：按现有格式保存原文；派生索引可生成外部 ID 和基础元数据。
- L1，低成本批处理：类型、关键词、实体、全文索引。
- L2，按需执行：语义向量、历史关联和手动分析报告。
- L3，定期执行：前一日日记总结、日报、周报、主题演化和跨周期综合。

## 6. `@` 的定位

保留现有 `@` 功能及其记录方式。它仍然表示用户正在向 AI 提问、咨询或要求即时执行一次任务，AI 完成后直接给出回答，并按现有格式将问题和回复写入当日日记。

示例：

```text
今天意识到，记录系统真正的价值可能不是记忆，而是发现自己反复绕回的问题。

@分析上面这个想法与我过去关于知识管理的记录有什么关系。
```

这个请求应像现在一样直接返回回答，而不是改造成“创建报告任务”。用户有时只是想咨询 AI，不能强迫所有 `@` 请求进入报告流程。

自动日记总结和分析报告是与 `@` 并列的系统能力，由定时任务或专门命令触发。`@` 调用的 AI 不再拥有修改日记 `<summary>` 的工具，避免即时问答意外改变日记总结。

## 7. 查阅功能的取舍

移除 `/q`。它与 `@` 的能力重叠，而且为了生成一句轻记录需要额外调用一次模型，长期价值不足。

当前保留两种已有方式：

- `/v [日期]`：直接查看原始日记，不经过 AI。
- `@问题`：需要 AI 检索、归纳或咨询时，使用现有问答入口；AI 可以调用历史读取和搜索工具。

是否在未来增加独立的语义查阅入口，等分析报告和索引形成真实使用反馈后再决定。本阶段不为命令形式提前设计复杂分类。

## 8. 数据分层设计

### 8.1 原始记录层

继续使用现有每天一个 Markdown 文件的形式，文件名、`<summary>` 区域、分隔线和原始记录内容格式均不改变。用户输入内容不被 AI 改写，也不在 Markdown 中增加稳定 ID 或其他隐藏标记。

未来如果需要稳定 ID，应由派生索引根据日期、顺序、内容哈希等信息维护，并提供重建规则。

### 8.2 派生索引层

建议使用 SQLite，第一阶段不需要引入独立数据库服务：

- `records`：记录 ID、时间、原文位置和内容哈希。
- `annotations`：类型、主题、实体、摘要、置信度和模型版本。
- `relations`：记录间的支持、反对、重复、延伸、演化关系。
- `jobs`：后台任务、优先级、状态、重试次数和错误。
- `reports`：报告类型、周期、输入快照、生成状态和路径。
- `feedback`：用户确认、否决、收藏和修正。
- FTS5：字面全文检索。

向量索引属于可重建缓存，必须记录嵌入模型和版本。可以先使用 SQLite 保存向量或轻量本地索引，等数据量证明有必要后再考虑专用向量数据库。

### 8.3 报告层

分析报告使用独立目录，不与原始日记混合：

```text
AnalysisReports/
  Daily/
  Weekly/
  Topics/
```

报告需要记录生成时间、覆盖周期、输入快照和模型版本。相同输入重复执行应覆盖同一版本或产生明确的新版本，不能无故生成重复报告。

### 8.4 代码架构

当前代码按五个有实际职责的模块拆分，`main.py` 只作为可执行入口：

```text
main.py
  └─ cli.py                 终端输入、命令解析和展示
       ├─ journal.py        原始日记格式、读写和检索
       ├─ ai_client.py      OpenAI 兼容请求、工具循环和联网搜索
       └─ analysis.py       总结、报告、调度和 Agent 编排
            ├─ journal.py
            └─ ai_client.py

settings.py                 各层共享的配置、目录和模型选择
```

边界规则：

- `journal.py` 是原始日记的唯一写入边界，分析 Agent 不得直接修改日记文件。
- `ai_client.py` 只处理 OpenAI 兼容协议和工具调用，不决定报告业务。
- `analysis.py` 负责准备分析上下文、编排模型或 Agent、验证结果并保存报告。
- `cli.py` 只负责把用户命令映射到应用能力，不承载分析提示词。
- `settings.py` 不依赖其他业务模块，避免循环依赖。

不按单个函数继续拆文件。未来只有当一个 Agent 具备独立提示词、输入输出结构、状态或测试时，才从 `analysis.py` 拆入 `agents/` 包。

### 8.5 长期分析对象与 Agent 路线

分享讨论进一步明确了长期数据链：

```text
Record → Thought → Hypothesis → Theme → Research → Insight → Report
```

- `Record`：不可改写的原始日记记录。
- `Thought`：从混杂日记中提取出的观点、问题、直觉和情绪。
- `Hypothesis`：可以被外部事实支持或反驳的判断。
- `Theme`：跨日期、跨记录持续形成的思想主题。
- `Research`：世界信息、历史类比、证据和反例。
- `Insight`：综合个人思想与外部世界后形成的候选洞见。
- `Report`：面向用户的最终结构化交付物。

后续 Agent 的建议接入顺序：

1. `Extractor`：从自然日记中提取 Thought，不要求用户改变记录方式。
2. `Explorer`：沿思想展开领域、假设、可能性和待验证问题。
3. `Cluster`：把长期碎片聚合为 Theme，识别演化和重复。
4. `Reviewer`：寻找逻辑漏洞、反例、遗漏变量和过度推断。
5. `World`：持续连接新闻、论文、产品和市场变化。
6. `Report`：汇总其他 Agent 结果，生成最终报告。

核心原则是“输入保持人性化，输出高度结构化”：用户负责方向、直觉和价值判断，AI 负责提取、展开、研究、关联和汇报。

## 9. 报告设计

### 日报

日报解决“今天出现了什么值得注意的变化”，不应只是逐条摘要：

- 今日重要思想与新问题。
- 自动形成或增强的主题。
- 与历史记录的少量强关联。
- 需要进一步研究或用户确认的内容。
- 当天自动或手动生成的分析报告位置。

### 周报

周报解决“这一周我的注意力和观点如何变化”：

- 本周主要思想簇及变化趋势。
- 新出现、持续强化或衰退的主题。
- 重复思考但仍未解决的问题。
- 观点之间的矛盾、转变和可能原因。
- 外部证据带来的支持或挑战。
- 下周最值得继续思考的少量问题。

### 专题报告

专题报告是后续可增加的手动分析类型，或由用户从已有报告中继续下钻产生；它不能改变 `@` 作为即时问答入口的语义。推荐结构：

1. 问题与结论摘要。
2. 用户已有观点及原始记录引用。
3. 历史关联与思想演化。
4. 外部事实、不同立场和反例。
5. AI 推断及不确定性。
6. 值得用户判断的问题或可验证的下一步。

报告必须明确区分“用户说过”“外部来源表明”“AI 推断”“AI 建议”。

## 10. 交互草案

```text
普通文字             按现有格式立即保存
@问题                按现有方式咨询 AI 并获得回复
/v [日期]            查看原始日记
/s [日期]            手动生成指定日期的日记顶部总结
/a daily [日期]      手动生成指定日期的分析日报
/a weekly [日期]     手动生成日期所在周的分析周报
```

`/q` 被删除，因此命令总数是减去一个、增加两个。自动任务与手动分析命令只显示开始、完成或失败状态；最终报告一次性展示，不持续刷出推理过程。`@` 仍按当前问答体验工作。

## 11. 工程改造路线

### 第一阶段：数据安全与可演进底座

- 为现有行为补最小测试，覆盖日期解析、追加、删除、总结替换和历史读取。
- 保持原始日记文件和内容格式不变；如需记录 ID，只在可重建的派生索引中维护。
- 建立 SQLite 元数据、FTS5 和任务表。
- 保留现有 `@` 问答在原始日记中的记录方式；自动分析报告必须与原始日记分离。
- 配置、原始日记、模型协议、分析编排和 CLI 已从 `main.py` 拆分，并保持清晰依赖方向。
- 将密钥迁移到环境变量或不入库的本地秘密配置。

验收标准：旧记录可无损读取；索引可完全重建；AI 失败不影响记录；现有主要命令仍兼容。

### 第二阶段：最小智库闭环

- 移除 `/q`，增加 `/s` 和 `/a`。
- 从 `@` 问答的工具列表中移除日记总结写入能力。
- 每天跨过零点后，自动总结前一天的原始日记并写入其 `<summary>` 区域。
- 在独立 `AnalysisReports/` 目录自动生成前一日日报。
- 每周自动生成上一完整自然周的周报。
- 支持手动重新生成指定日期的总结、日报和周报。

验收标准：原始日记格式与现有 `@` 体验不变；跨日后前一天总结自动更新；日报和周报出现在独立目录；手动命令可重复生成。

### 第三阶段：高阶关联与反馈

- 增加语义召回和主题聚类。
- 建立支持、反对、重复、延伸、演化关系。
- 让用户确认或否决 AI 关联，并用于后续排序。
- 建立长期主题档案和专题报告。
- 根据实际使用数据调整自动研究的触发条件和成本预算。

验收标准：系统能够发现仅靠关键词搜索找不到的强关联，并且引用准确、噪声可控。

## 12. 暂不建议优先实现

- 多 Agent 自由讨论。当前更需要可靠的数据边界、检索和报告闭环。
- 一开始就部署专用向量数据库。个人记录规模下 SQLite 足够作为第一阶段基础。
- 自动把所有想法变成任务。思想记录与任务管理应有关联，但不能混为一体。
- 让 AI 自动改写原文或替用户确认观点。
- 复杂 GUI。应先验证后台加工和报告是否真正提高思考质量。

## 13. 待讨论决策

1. 自动分析任务失败后采用何种重试间隔和提醒方式？
2. 用户确认的长期观点是否需要独立的“观点卡片”？
3. 日报默认应该更偏向当天内容，还是始终与全部历史记录做关联？
4. 联网研究的成本、隐私范围和自动触发条件如何配置？

这些问题不会阻塞第一阶段的数据底座，但会影响第二阶段的具体交互。
