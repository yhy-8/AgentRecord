# AGENTS.md

本文件规定 AI 编码代理在本仓库中的工作方式。它是代码修改规范，不是产品设计文档。

## 1. 编码前先理解

**不要猜测，不要掩盖不确定性，先说明关键取舍。**

开始实现前：

- 阅读与任务相关的代码、配置、README 和现有数据格式。
- 明确需求、假设、成功标准和兼容性影响。
- 存在多种合理理解时，说明差异；如果差异会显著影响结果，先询问用户。
- 如果有更简单的方案，主动提出。
- 不理解的地方应明确指出，不能用臆测补全。

## 2. 简单优先

**只写解决当前问题所需的最少代码。**

- 不实现用户没有要求的功能。
- 不为单次使用创建抽象层。
- 不提前加入没有现实用例的扩展点或配置项。
- 不为实际上不可能发生的场景堆叠错误处理。
- 如果实现明显可以更短、更直接，应先简化再交付。

较大的架构升级应拆成可独立验证的小步骤，避免一次性重写整个项目。

## 3. 精准修改

**只修改完成任务必需的部分，只清理本次改动产生的问题。**

- 不顺手重构、重排或格式化无关代码。
- 保持现有代码风格，除非任务本身要求统一风格。
- 发现无关的坏味道、死代码或缺陷时，可以在交付说明中指出，但不要擅自修改。
- 删除由本次改动导致的无用导入、变量、函数和文件。
- 尊重工作区中已有的用户改动，不覆盖、不回退、不删除来源不明的变更。

每一处变更都应能直接追溯到用户需求。

## 4. 目标驱动与验证

**把请求转化为可以检查的完成条件，并循环到验证通过。**

例如：

- “修复日期解析” → 增加可复现用例，再让用例通过。
- “增加数据迁移” → 验证旧格式、迁移结果、重复运行和失败恢复。
- “拆分模块” → 对比拆分前后的主要行为和打包结果。

多步骤任务应维护简短计划，并在完成前确认：

- 相关测试、静态检查或最小运行检查已经执行。
- 代码、README、配置示例和帮助文本描述一致。
- 未能验证的部分及原因已明确告诉用户。
- 没有把“代码已写完”误当成“目标已实现”。

## 5. AgentRecord 项目约束

- 原始记录属于用户数据。涉及记录写入、删除、迁移和 AI 修改权限时，优先保证不丢失、不静默覆盖且可恢复。
- 保持现有 Markdown 记录的兼容性；格式变更必须提供迁移和验证方案。
- API 密钥和个人记录不得出现在提交、测试输出或异常日志中。
- 网络请求必须设置超时，并给出清晰的失败状态；失败不得破坏已经保存的原始记录。
- 保持 Windows 与类 Unix 环境兼容，注意路径、UTF-8、终端输入和 ANSI 行为。
- 修改依赖或入口代码后，应考虑 GitHub Actions 与 PyInstaller 打包行为。
- 遵守现有模块边界：`journal` 管原始日记，`ai_client` 管 OpenAI 兼容协议与工具循环，`analysis` 管分析编排与调度，`cli` 管终端交互，`settings` 管配置。不要让 Agent 直接操作终端或绕过 `journal` 写原始日记。
- 新分析能力先接入 `analysis`；只有当某类 Agent 形成独立状态、输入输出和足够实现后才拆入 `agents/` 包，避免创建功能过少的文件。

## 6. 完成标准

这些规范生效的表现应当是：改动范围更小、假设更透明、数据更安全、验证结果更明确，并且不会因为过度设计造成额外返工。
